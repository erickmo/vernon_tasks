ADR-020 — UI composition strategy for the desktop portal Projects module (P3.1)

ADR-020 — Portal Projects reuses Portal OKR composition (Approach A) — vanilla React Query + composed UI

Status: Accepted Date: 2026-05-17 Spec: Portal Projects P3.1 PRD Related: ADR-019
Context

The P3.1 portal Projects module needs the same shape as P2 OKR: master–detail list, multi-criteria filtering, bulk row selection, inline status/PDCA edits, and a create/edit form. It additionally renders a compact linked-Objective summary inside the project detail. The decision is whether to (a) repeat the OKR composition pattern, or (b) extract a generic master-detail abstraction (MasterDetailLayout, generic table/filters/bulk-actions) now that we have a second instance.

Decision

Adopt Approach A — same vanilla React Query + composed primitives as P2 OKR. Reuse: the existing components/ Table/Button/Input/EmptyState/PageLayout/PageSkeleton; react-hook-form + zod from P2; the useObjective hook from pwa/src/portal/okr/hooks/ (no copy) to power the linked-OKR preview. Do not extract a shared MasterDetailLayout / GenericTable / GenericFiltersBar abstraction yet — defer until a third domain (Workforce or Reports) demonstrates a real third instance and a real pain point.

Consequences
  • + Zero new top-level dependency; stays within the projects chunk ≤120 KB gz budget.
  • + Cross-domain consistency — Projects looks and behaves like OKR; managers learn once.
  • + Cross-domain reuse is concrete (useObjective consumed directly), validating the loose-coupling model.
  • − Mild duplication across portal/okr/FiltersBar.tsx ↔ portal/projects/FiltersBar.tsx, same for Table + BulkActions. Accepted; revisit when a 3rd domain ships (consolidation rule of three).
  • − No free virtualization — project lists are bounded (≤500 per page server-side) so this is not a near-term concern.
Alternatives
  • Extract shared MasterDetailLayout + GenericTable + GenericFiltersBar now — rejected as premature. Two instances are not enough signal to design a correct abstraction; the cost of a wrong abstraction is higher than the cost of a third near-duplicate. Re-evaluate at P4 (Workforce).
  • TanStack Table v8 — column models, headless, virtualizable. Rejected for P3.1 for the same reasons as ADR-019: bounded scale (<100 rows per page typical, 500 max), and consistency with the OKR module.
  • AG Grid Community — heavy bundle, license surface for enterprise features; rejected.
