ADR-001 — Build on Frappe Framework

Status: Accepted Date: 2026-04 (project inception)
Context

Company-wide tool integrated with existing ERPNext stack. Users, roles, permissions, audit, email already in Frappe. Greenfield = duplicate work.

Decision

Build as Frappe v15 app. Reuse User, Notification Log, permissions, scheduler, Report Builder, REST. Add domain doctypes only.

Consequences
  • + Zero auth/role work; native CRUD via /api/resource/.
  • + Built-in workspaces, reports, fixtures, scheduler.
  • − Tied to Frappe upgrade cadence and Python 3.11+.
  • − Dated Desk UX → motivates PWA (ADR-002).
Alternatives
  • Django + DRF: rebuild auth + roles + reporting.
  • SaaS (Asana, Linear): no OKR/KPI integration; no PDCA model.

ADR-002 — Dual UI: Frappe Desk + Mobile PWA

Status: Accepted Date: 2026-05-11
Context

ICs work on phones; leaders + execs on desktop with analytics. Single UI fights both. Frappe Desk = power users; bad mobile-first.

Decision

Keep 7 Frappe Desk pages for desktop power users + leaders/execs. Ship installable React PWA at /m/ for mobile-first IC and leader review.

Consequences
  • + Best UX per surface; backend single source of truth.
  • − Double surface to test (Vitest/Playwright + bench tests).
  • − Risk of UI drift; mitigated by responsive nav (ADR-016).

ADR-003 — React 18 + Vite 5 + TypeScript 5 for PWA

Status: Accepted
Context

Need fast HMR, small prod bundle, mature ecosystem, type safety.

Decision

React 18 + Vite 5 + TS 5. react-router-dom@6, @tanstack/react-query@5, idb-keyval, Workbox 7, recharts (lazy).

Consequences
  • + ~300 ms dev reload; ~95 KB gz main bundle; Concurrent Suspense.
  • − Adds Node tooling step to deploy pipeline.
  • − Recharts cost mitigated by lazy split (ADR-011).
Alternatives
  • Vue/Svelte: smaller ecosystem for Frappe consultants.
  • Next.js: SSR not needed; SPA served by Frappe.
  • Preact: bundle win but loses React-ecosystem libs.

ADR-004 — Workbox StaleWhileRevalidate offline strategy

Status: Accepted
Context

Mobile users hit spotty networks. Need fast first paint + fresh online + readable offline. Full offline-first w/ sync queue = high complexity, hard write conflicts.

Decision

Workbox 7 generateSW. Precache app shell. Runtime StaleWhileRevalidate on /api/method/vernon_tasks.* with 1-day cache expiry. Mutations rejected when offline; UI disables buttons + toast.

Consequences
  • + Cached lists render instantly; fresh fetched in background.
  • + Offline reads always work; no conflict-resolution complexity.
  • − No queued offline writes (deferred).
  • − 1-day staleness window; mitigated by SWR refresh.

ADR-005 — Reuse Frappe session cookie (no JWT)

Status: Accepted
Context

PWA hosted same-origin as Frappe (/m/). Separate JWT = duplicate user store, refresh-token rotation, key mgmt. Frappe already issues httpOnly sid cookie.

Decision

PWA logs in via POST /api/method/login. Receives sid cookie. All subsequent API calls send cookie automatically. CSRF token fetched via /m/_boot; required on mutations.

Consequences
  • + Zero auth surface; native Frappe role checks work as-is.
  • + Single sign-out across Desk + PWA.
  • − Same-origin only; no third-party PWA host.
  • − Session expiry → relogin modal (preserves route state).

ADR-006 — TanStack Query 5 for client cache

Status: Accepted
Context

Need declarative server-state cache with SWR, optimistic updates, invalidation.

Decision

Use @tanstack/react-query@5 as sole client-cache layer. Query keys: ["my-work"], ["task", id], ["search", q], etc. Stale time 5 min. Invalidate on mutation success.

Consequences
  • + No Redux/Zustand boilerplate; UI state via React local state.
  • + Optimistic updates via onMutate + onError rollback.
  • − Two cache layers (RQ in-memory + Workbox SW); coordination via SWR semantics.
