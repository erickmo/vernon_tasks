ADR-022 — Prefer Frappe REST CRUD; move domain rules into doctype controllers, not custom whitelisted wrappers

ADR-022 — REST-first, hooks-for-logic

Status: Accepted Date: 2026-05-25 Scope: www-react SPA → vernon_tasks backend Implementation status: Partial — VT Brand + VT Project single-doc CRUD shipped; aggregations remain custom

Context

The www-react SPA originally used custom whitelisted endpoints (/api/method/vernon_tasks.brand.api.portal_brands.*, /api/method/vernon_tasks.task.api.portal_projects.*) for basic CRUD on VT Brand and VT Project. Most of those endpoints were thin wrappers around frappe.get_doc / insert / save / delete_doc with a small amount of domain logic (field whitelisting, FK integrity check, rename on PK change).

This duplicates code Frappe already provides and adds a maintenance surface that drifts from the doctype itself.

Decision

  1. REST-first. Single-document CRUD MUST use Frappe's built-in REST endpoints (/api/resource/{DocType} with GET / POST / PUT / DELETE). The frontend client maps the Frappe primary key name to our domain id at the boundary.
  2. Hooks-for-logic. Domain rules that previously lived in the wrapper move into the doctype controller (validate, on_trash, autoname, permlevel) so the rule fires regardless of how the document is mutated (REST, scripted, fixture import, child relationship).
  3. Custom endpoints only for orchestration. Endpoints that aggregate across doctypes, perform multi-document batch operations, or compute derived data (e.g. list_projects with blocked_count, get_project_detail, bulk_move_tasks, get_burndown) stay on /api/method/....

Concrete changes

ConcernBeforeAfter
Brand CRUD (create/read/update/delete/list/search) /api/method/...portal_brands.* /api/resource/VT Brand via brandsApi.ts

Brand rename on brand_name change Manual frappe.rename_doc in update_brand Doctype autoname: field:brand_name — REST PUT auto-renames
Project single-doc CRUD /api/method/...portal_projects.create/update/delete /api/resource/VT Project via projectsApi.ts

Project list / detail / bulk ops / OKR enrich / burndown Custom /api/method/... endpoints Unchanged — kept custom (aggregation + multi-doc orchestration)
Legacy vernon_tasks/api/{projects,sprints}.py Used by old pwa/ SPA Unchanged — retire together with pwa/

Consequences

Positive

  • Less Python wrapper code to maintain; doctype is the single source of truth.
  • Domain rules fire from any mutation path (REST, fixtures, scripted, child propagation).
  • Frontend benefits from standard Frappe REST filter operators, pagination, ordering.
  • Permission enforcement comes free from Frappe's role/permlevel system.

Negative / trade-offs

  • Frappe REST responses use name (PK) — frontend must map to id at the API boundary (one helper per resource).
  • Capability flags (can_create / can_write / can_delete) are not natively returned in a single REST call; we still ship a small get_*_permissions custom endpoint for the portal's optimistic UI gating.
  • Field-level whitelisting now relies on permlevel + role permissions rather than a Python dict — protected fields must be declared in the doctype JSON, not in code.

Anti-patterns this ADR rules out

  • Creating a new @frappe.whitelist() wrapper just to rename response fields. Map at the frontend boundary instead.
  • Re-implementing Frappe's link integrity check (check_if_doc_is_linked) inside a custom delete endpoint.
  • Calling frappe.rename_doc manually for doctypes whose autoname is field:<field>.
