ADR-012 — Allowlisted client telemetry, 90-day retention

Status: Accepted
Context

Need observability for rollout (error rate, install rate, funnel) without external SaaS (privacy + cost). Free-form events risk PII leakage.

Decision

Client calls task.api.telemetry.log_event(event, props). Server enforces allowlist of ~40 event names. Rate-limit 60/min/user. Storage in Vernon Telemetry Event doctype, daily purge ≥90 days. Report Builder for funnels.

Consequences
  • + All data stays in Frappe; no third-party.
  • + Allowlist prevents accidental PII fields.
  • − No real-time dashboards (ops use Report Builder).

ADR-013 — Pre-rollout hardening via security.py guards

Status: Accepted
Context

Before company-wide rollout (CSO sign-off), PWA APIs unbounded → potential abuse (brute mutations, large payloads, clickjack on SPA shell).

Decision

Add task/api/security.py with reusable guards: require_login, rate_limit, clamp_int, max_str. Apply per-endpoint table. Inject CSP / X-Frame / Referrer / Permissions headers on /m/* responses.

Consequences
  • + Defense in depth additive layer; no business logic change.
  • + Reusable in future endpoints.
  • − Slight per-request overhead (Redis incr). Acceptable.

Out of scope: WAF, DocType permission audit, frontend validation.

ADR-014 — Phased PWA rollout P0.5 → P4b behind boot flags

Status: Accepted
Context

Big-bang PWA = high risk; pilot data needed before company-wide. Need backout per feature.

Decision

Sequential additive phases: P0.5 → P1a → P1b → P2 → P3a → P3b → P4a → P4b. Each phase behind feature flag exposed via /m/_boot. Pilot gates: error_boundary < 1 %, login_failure < 5 %, install rate ≥ 30 %, no P0 bugs.

Consequences
  • + Clear ship/measure/decide loop. Per-feature backout via flag.
  • − Coordination overhead per phase; mitigated by spec+plan template.

ADR-018 — Git-SHA service-worker cache key

Status: Accepted
Context

Service workers infamously cache stale assets after deploy. Manual cache versioning forgotten → users stuck on old code.

Decision

Vite build injects VITE_GIT_SHA into SW cache name. Workbox generateSW uses it as cacheId. Every deploy = new SHA = new cache namespace; old cache evicted on activate.

Consequences
  • + Cache-busting automatic. Rollback possible: revert SHA → SW serves prior cache.
  • − Slight cold-load cost on every deploy (one fresh fetch).
