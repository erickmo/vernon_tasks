ADR-019 — UI composition strategy for the desktop portal OKR module (P2)

ADR-019 — Portal OKR uses vanilla React Query + composed UI (Approach A) over TanStack Table

Status: Accepted Date: 2026-05-17 Spec: Portal OKR P2 PRD
Context

The P2 portal OKR module needs a master–detail list, multi-criteria filtering, bulk row selection, inline editing on Key Results, and create/edit forms. Two stack options:

  • A — Vanilla React Query + composed primitives (existing components/ Table, Button, Input, EmptyState).
  • B — TanStack Table v8 + column models (purpose-built for sortable, filterable, paginated, virtualized grids).
Decision

Adopt Approach A. Expected scale is <100 Objectives per active period; virtualization and column-model abstractions are not needed at this volume. The shared components/Table and react-query already cover the load. Reserve Approach B as an upgrade path if pain becomes real (≥500 rows, perceptible jank, or column-model reuse across modules).

Consequences
  • + Zero new top-level dependency; smaller bundle (≤120 KB gz target for the okr chunk).
  • + Style consistency with mobile PWA — single component library, single design language.
  • + Easier for new contributors — no column-model API to learn.
  • − Hand-rolled sort/filter/bulk-select logic in ObjectiveTable.tsx; risk of duplication if Projects/Workforce sub-PRDs need the same.
  • − No free virtualization — will need to revisit at ≥500 rows.
Alternatives
  • TanStack Table v8 — column models, headless, virtualizable. Rejected for P2 due to scale fit and bundle cost; promoted to candidate if P3/P4 modules require shared grid abstractions.
  • AG Grid Community — feature-rich but heavy bundle and license surface area for enterprise features; rejected.
  • Material React Table — pulls MUI; conflicts with existing design system; rejected.
