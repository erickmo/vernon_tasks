ADR-021 — Build the new desktop dashboard as a standalone Vite + React SPA on its own domain

ADR-021 — Standalone www-react SPA on separate domain

Status: Accepted Date: 2026-05-23 Spec: www-react Dashboard Design Related: ADR-002, ADR-005
Context

We need a desktop-first dashboard that serves IC + Leader + Exec audiences with a single role-aware UI: weekly worksheet planner, project-level OKR/KR drill-down, bulk task actions, and scheduled exports. We already ship two frontends — Frappe Desk pages and the /m/* mobile PWA. The pwa/src/portal/* tree under /portal/* on the Frappe origin grew organically out of the PWA bundle and is coupled to Frappe's asset pipeline, the Workbox service worker, and the same-origin session-cookie path. Extending it for the new desktop dashboard would mean further tangling desktop and mobile concerns in one bundle and one deploy artifact.

Decision

Build the new desktop dashboard as a standalone Vite + React 18 + TypeScript SPA in www-react/, deployed to its own domain (e.g. dashboard.vernon.local) via Caddy reverse-proxy. Auth reuses the Frappe session cookie cross-domain (CORS allowlist + SameSite=None; Secure) — no JWT, consistent with ADR-005. The SPA depends on no Frappe asset hooks; it ships static dist/ served by Caddy. Stack: React Router 7, TanStack Query 5, Tailwind, dnd-kit, recharts, sonner, react-i18next, Vitest + Playwright. Caddy snippet lives at caddy/dashboard.Caddyfile.

Consequences
  • + Clean stack — no Frappe asset pipeline coupling; npm run build is the entire build.
  • + Single role-aware SPA for IC + Leader + Exec — no per-role bundle split, no Desk context switch.
  • + Future-proof: dashboard scales independently of Frappe (CDN, separate cache, separate rollout).
  • + Brand differentiation — own domain, own theme, no Desk chrome leaking in.
  • + Reuses Frappe session cookie (ADR-005) — no new auth surface.
  • − Cross-domain auth complexity: requires allow_cors in common_site_config.json, SameSite=None; Secure cookie, mandatory TLS in all environments.
  • − Duplicate base-layer components vs pwa/src/portal/* (buttons, tables, form primitives). Accepted; consolidation deferred until either side stabilises.
  • − Two deploy artifacts (Frappe bench + Caddy static) and two build pipelines.
  • − Two SPAs with overlapping /portal/* route names on different origins — operational documentation must call this out (see portal docs § www-react).
Alternatives
  • Extend pwa/src/portal/* under the Frappe origin — rejected. Couples desktop dashboard to Workbox service worker, mobile breakpoints, and Frappe's static-asset hooks; widens an already-large bundle; ties dashboard rollout to the PWA release train.
  • New Frappe Desk pages — rejected. Desk's chrome and form-builder model do not fit a custom dnd planner / role-aware landing; would regress the desktop UX vs current Desk pages.
  • Server-rendered Frappe web pages (Jinja) — rejected. Too dynamic for SSR; would lose TanStack Query caching and dnd ergonomics.
  • Same-origin subpath (/dashboard/* via Frappe) — rejected. Avoids the CORS surface but reintroduces the asset-pipeline coupling we are trying to escape.
