2.2 Request lifecycle (PWA mutation)

  1. User taps "Selesai" on task card.
  2. useUndoableMutation shows 5s undo toast (no API call yet).
  3. After 5s without cancel → React Query onMutate applies optimistic update.
  4. fetch('/api/method/vernon_tasks.task.api.my_work_mutations.complete', {...}) with cookie + X-Frappe-CSRF-Token.
  5. Frappe routes → page handler → require_login() → rate_limit("complete", 30) → service.
  6. Service mutates VT Task → on_submit triggers point_calculator.calculate_points → row in Task Point Log → User Point Summary update.
  7. Response returns. onSuccess invalidates ["my-work"] query.
  8. If error → onError rollback optimistic state + toast.

4. Data Flow Diagrams

4.1 Task completion → points

PWA / Desk → my_work_mutations.complete(task_id)
            └─→ security.require_login + rate_limit("complete", 30)
            └─→ frappe.get_doc("VT Task", task_id).db_set("pdca_phase", "DONE")
                └─→ doc_event on_submit
                    └─→ point_calculator.calculate_points(task)
                        ├─→ base = weight × multiplier
                        ├─→ early_bonus or late_penalty
                        ├─→ revision_deduction
                        └─→ Task Point Log row(s) inserted
                            └─→ User Point Summary recomputed (or +=)
            └─→ analytics.invalidate_project_cache(project)
            └─→ Notification Log inserted (assignee / leader)
                └─→ after_insert → push_sender.send_push_for_notification
                    └─→ filter by Vernon Push Preference
                    └─→ POST to each Vernon Push Subscription endpoint
                        └─→ dead endpoint (404/410) → prune row

4.2 Boot & offline read

PWA load → GET /m/ → www/m.py renders shell + injects CSP headers
        ↓
SW activate (cache name = "vt-{git_sha}")
        ↓
React mounts → GET /m/_boot → {user, roles, csrf_token, flags}
        ↓
useQuery(["my-work"]) → fetch /api/method/...my_work
   ├─ online  → SW StaleWhileRevalidate: serve cache, fetch fresh, update cache
   └─ offline → SW returns cached response
                ↓
              OfflineBanner shown; StaleBadge if >1h old

6. API Contracts

6.1 Conventions

  • Base URL: POST /api/method/vernon_tasks.<module>.<file>.<fn>
  • Auth: cookie (sid) or Authorization: token <key>:<secret>
  • CSRF: X-Frappe-CSRF-Token on mutations (browser context)
  • Body: application/x-www-form-urlencoded or JSON
  • Response: { "message": <payload> }
  • Errors: HTTP 400 (ValidationError), 403 (PermissionError), 417 (rate-limit / bound), 500

6.2 Sample contracts

my_work.list

request:  POST /api/method/vernon_tasks.task.api.my_work.list
response: {
  "message": {
    "overdue":  [TaskCard],
    "today":    [TaskCard],
    "upcoming": [TaskCard]
  }
}

TaskCard = {
  id: string, title: string, status: string,
  priority: "Tinggi"|"Sedang"|"Rendah",
  due_date: "YYYY-MM-DD", project: string, points: number
}

my_work_mutations.complete

request:  { task_id: string }
response: { ok: true, idempotent?: boolean, task_id: string }
errors:   403 require_login · 417 rate_limit · 400 invalid state

my_work_mutations.log_progress

request:  { task_id, hours: 0.25..8 (step 0.25), note?: string (≤1000) }
response: { ok: true, actual_hours: number }
errors:   417 rate_limit / clamp · 400 invalid hours

analytics.get_burndown

request:  { sprint: string }
response: {
  labels: ["YYYY-MM-DD", ...],
  ideal:  [number, ...],
  remaining: [number, ...],
  unestimated_count: number
}

exec_analytics.get_health_score

response: {
  score: 0..100,
  okr_pct: number,
  ontime_pct: number,
  velocity_health: number,
  breakdown: { okr_weight: 0.4, ontime_weight: 0.3, velocity_weight: 0.3 }
}

push.subscribe

request:  { endpoint, p256dh, auth, user_agent }
response: { ok: true, renewed: boolean }
errors:   417 rate_limit (5/min) · max_str(endpoint, 2048)

8. Service Worker & Cache

8.1 Strategy

  • Workbox 7 generateSW via vite-plugin-pwa.
  • Precache app shell (HTML, JS, CSS, icons) — 5 entries.
  • Runtime: StaleWhileRevalidate on /api/method/vernon_tasks.*, max-age 1 day, max 50 entries.
  • Cache name: vt-{VITE_GIT_SHA}. Old caches purged on activate.
  • POST mutations bypass cache; fail with network error if offline.

8.2 Update flow

  1. Deploy → SW fetches new sw.js on next page load.
  2. install → precache new shell.
  3. activate → delete caches not matching vt-{new_sha}.
  4. Client reloads (no skipWaiting in current config; reload on next visit).

9. Auth & Session

9.1 Login flow

User → POST /api/method/login
       body: usr=<email>&pwd=<password>
       ← Set-Cookie: sid=...; HttpOnly; SameSite=Lax
User → GET /m/_boot
       cookie: sid=...
       ← { user, roles, csrf_token, flags }
       (stored: user/roles → sessionStorage; csrf_token → memory)
User → any mutation
       cookie + header X-Frappe-CSRF-Token

9.2 Session expiry

  • Any 401/403 from API → ReloginModal opens.
  • Current route + state preserved in modal state.
  • After successful relogin → modal closes, original mutation retried.

12. Observability

12.1 Telemetry pipeline

  • Client emits logEvent(event, props) → POST /api/method/...telemetry.log_event.
  • Server validates event name against allowlist; truncates props ≤ 2048 bytes; rate-limits 60/min/user.
  • Stores in Vernon Telemetry Event: user, event, props (JSON), ts.
  • Daily scheduler purge_old_telemetry deletes rows > 90 days.
  • Funnel analysis via Frappe Report Builder reports.

12.2 Required reports (pre-launch)

  1. PWA Boot Funnel — pwa_boot → login_success → page_view
  2. Error Rate — error_boundary count / page_view count
  3. Install Funnel — install_prompt_shown → install_accepted
  4. Mutation Health — task_complete count + failure rate
  5. Push Delivery — push_subscribed vs push_received

12.3 Server logs

  • Frappe error log captures stack traces; review daily during pilot.
  • nginx access logs for /m/* — monitor 4xx/5xx ratio.

13. Infrastructure & Deployment

13.1 Topology

  • Standard frappe-bench on Ubuntu 22.04+.
  • MariaDB 10.6+, Redis (queue + cache), nginx → gunicorn workers.
  • Node 18+ on build host (PWA build).

13.2 Deploy steps

cd ~/frappe-bench
git -C apps/vernon_tasks pull
cd apps/vernon_tasks/pwa && npm ci && npm run build
cd ~/frappe-bench
bench --site <site> migrate          # apply schema changes, run patches
bench --site <site> clear-cache
bench restart                          # gunicorn + workers

Service worker auto-busts via git-SHA cache name.

13.3 VAPID keys (one-time)

bench --site <site> execute vernon_tasks.task.api.push.cli_generate_vapid
# stores public + private keys in VT Settings

13.4 Rollback

  1. git checkout <prev SHA> in apps/vernon_tasks.
  2. Rebuild PWA, bench restart.
  3. SW activates old cache namespace; users get prior code.
  4. If schema migration ran → forward-fix in next patch; never rollback migrations destructively.

15. Coding Standards

15.1 Python

  • Layer separation: Controller (validation, db_set) · Service (logic) · Page Handler (auth + IO).
  • Function length ≤ 40 lines; class < 300 lines / 5 responsibilities.
  • Named constants module-level; no magic strings/numbers.
  • DI: services take settings/config as params.
  • ignore_permissions=True only in internal services / scheduler.
  • Email sends wrapped in try/except (silent skip).
  • Prefer frappe.get_all() / frappe.get_doc() over raw SQL.

15.2 TypeScript

  • Strict mode; no any.
  • API payload types in pwa/src/api/types.ts shared across modules.
  • Components ≤ 150 lines; extract hooks for logic.
  • Inline styles + CSS variables (no styled-components / Tailwind).
  • id-ID strings centralized in pwa/src/i18n.ts.
