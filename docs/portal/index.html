What is Vernon Tasks?

Vernon Tasks is a Frappe app that unifies daily work execution (PDCA), project management (Agile sprints), and performance management (OKR) into a single system accessible via Frappe Desk and an installable mobile PWA.

Who Uses It?

Individual Contributor (IC)PWA / Desk My WorkPlan tasks, log hours, submit for review
Team LeaderPWA / Desk Leader ReviewApprove/reject tasks, review sprint burndown
ExecutiveDesktop Portal / Desk Exec AnalyticsOKR roll-up, KPI trends, Health Score
AdminFrappe Desk settingsConfigure roles, push keys, VT settings

Quick Start

Install

cd frappe-bench
bench get-app https://github.com/your-org/vernon_tasks
bench --site mysite install-app vernon_tasks
bench --site mysite migrate

Build the PWA

cd apps/vernon_tasks
bash build-pwa.sh

Configure Push Notifications (VAPID)

bench --site mysite execute vernon_tasks.task.api.push.generate_vapid_keys

Paste the output keys into VT Settings doctype.

Local Dev

cd apps/vernon_tasks/pwa
npm install
npm run dev   # Vite dev server on :5173, proxies to bench

Run Tests

# Backend
bench --site mysite run-tests --app vernon_tasks

# PWA
cd apps/vernon_tasks/pwa && npm test

# e2e (needs running site)
npm run e2e

Architecture

Layer Responsibility

LayerLocationResponsibility
Handler / Controllertask/api/*.pyParse input, validate format, call service, format output. No business logic.

Domain / Entitytask/doctype/*Core business rules, PDCA state machine, points invariants.
Infrastructuretask/api/push*.pyExternal services: Web Push, telemetry.

Key Services

ServicePurpose
point_calculator.pyPoints = complexity × difficulty factor, revision deductions, leader override
scheduling_engine.pyCapacity-aware scheduling, recurring task spawning
burndown_service.pySprint burndown with ideal vs actual
forecast_service.pySprint completion forecast via velocity
risk_evaluator.pyAt-risk task detection (late, blocked, scope drift)
health_score_service.pyHealth = OKR 0.4 + ontime 0.3 + velocity 0.3
push_sender.pyVAPID push via WebPush, subscription pruning

Security Patterns

  • All API endpoints use @frappe.whitelist()
  • Rate limiting, input clamping, max_str via task/api/security.py guards
  • Role checks at endpoint top via frappe.has_role()
  • CSP / X-Frame-Options headers on /m/* via www/m.py
  • Telemetry purged after 90 days via scheduler

Key ADRs

  • ADR-001 Build on Frappe Framework (not standalone Django)
  • ADR-002 Dual UI: Frappe Desk + Mobile PWA
  • ADR-005 Reuse Frappe session cookie (no JWT)
  • ADR-011 Security guards centralised in security.py
  • ADR-016 Dual-navbar PWA layout with primary gradient

Notification Dropdown

  • Rendered by NotificationDropdown inside TopNav; triggered by bell button.
  • Unread indicator: filled primary dot + 3px primary border-left on item row.
  • Read indicator: outlined grey dot, transparent border-left.
  • Unread item background: C_PRIMARY_L (#f3eeff).
  • Bell button when open: C_NAV_ACTIVE background, rgba(255,255,255,0.30) border.
  • Fetches up to 10 latest notifications on open; stale after 30s.
  • "Mark all read" button appears only when unread items exist.

Avatar

  • Background always C_PRIMARY (#7c4dab) — no per-user colour randomisation.

PDCA-driven work unit. Core of Vernon Tasks.

Modules

  • task/doctype/ — vt_task, task_dependency, task_schedule_entry, task_point_log, recurring_rule

PDCA State Machine

BACKLOG → PLAN → DO → CHECK → DONE / ACT

www-react Dashboard SPA

Standalone Vite + React 18 SPA shipped 2026-05-23. Runs on its own domain (e.g. dashboard.vernon.local) and talks to Frappe API across origins. Not part of the Frappe asset pipeline — built independently and served by Caddy.

Hosting Model

  • Build artifact: www-react/dist/ (static files).
  • Reverse-proxied by Caddy → see caddy/dashboard.Caddyfile.
  • SPA fallback: all non-asset routes → index.html (React Router 7 handles client routing).

Cross-Domain Authentication

  • Login posts usr/pwd to POST {VITE_API_BASE}/api/method/login with credentials: 'include'.
  • Frappe site must set: "allow_cors": "https://dashboard.vernon.local" in common_site_config.json.
  • Session cookie response must include SameSite=None; Secure; configure Frappe accordingly (TLS required).
  • All API calls go through @/lib/api (axios + withCredentials) — no JWT.

Differences vs existing /portal/* (in-Frappe)

The existing pwa/src/portal/ tree is served from the Frappe site under /portal/* on the same origin. The new www-react SPA also uses a /portal/* path prefix but on a different domain. They are two separate React applications with overlapping route names but no shared code or build pipeline. Migration / consolidation is out of scope for the 2026-05-23 ship.

User

My Work — Daily Queue

My Work is your primary workspace. It shows your tasks for today, lets you log progress, and sends tasks for leader review.

Access

  • Desk: My Tasks workspace → My Work page
  • PWA: Open https://yoursite/m/ on mobile or desktop

Walkthrough

  1. 1

    Open My Work

    Your tasks are sorted by due date. Each card shows PDCA phase, points, and project.

  2. 2

    Move a Task to PLAN

    Tap the task → tap Move to PLAN. Add notes on your approach.

  3. 3

    Work on the Task (DO)

    Tap Start Work to move to DO. Log hours as you progress.

  4. 4

    Submit for Review (CHECK)

    When done, tap Submit for Review. Your leader receives a push notification.

  5. 5

    Task Approved → DONE

    Once approved, the task moves to DONE and points are credited.

Snooze & Undo

Swipe left on a task to reveal Snooze (defer to tomorrow) and Undo (revert last action). Both are reversible.

Push Notifications

VAPID Setup

bench --site mysite execute vernon_tasks.task.api.push.generate_vapid_keys

Copy the output keys into VT Settings → VAPID Public Key and VAPID Private Key.

⚠
Regenerating VAPID keys invalidates all existing push subscriptions. All users must re-subscribe. Only regenerate if the private key is compromised.

Push Pipeline

  • Event fires (task submitted, approved, etc.)
  • Frappe hook calls push_sender.send_push_notification
  • Checks per-user preferences via push_prefs.py
  • Sends via WebPush library to stored subscriptions
  • Failed / expired subscriptions pruned automatically
