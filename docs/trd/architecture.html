2.2 Request lifecycle (PWA mutation)

  1. User taps "Selesai" on task card.
  2. useUndoableMutation shows 5s undo toast (no API call yet).
  3. After 5s without cancel → React Query onMutate applies optimistic update.
  4. fetch('/api/method/vernon_tasks.task.api.my_work_mutations.complete', {...}) with cookie + X-Frappe-CSRF-Token.
  5. Frappe routes → page handler → require_login() → rate_limit("complete", 30) → service.
  6. Service mutates VT Task → on_submit triggers point_calculator.calculate_points → row in Task Point Log → User Point Summary update.
  7. Response returns. onSuccess invalidates ["my-work"] query.
  8. If error → onError rollback optimistic state + toast.

4. Data Flow Diagrams

4.1 Task completion → points

PWA / Desk → my_work_mutations.complete(task_id) └─→ security.require_login + rate_limit("complete", 30) └─→ frappe.get_doc("VT Task", task_id).db_set("pdca_phase", "DONE") └─→ doc_event on_submit └─→ point_calculator.calculate_points(task) ├─→ base = weight × multiplier ├─→ early_bonus or late_penalty ├─→ revision_deduction └─→ Task Point Log row(s) inserted └─→ User Point Summary recomputed (or +=) └─→ analytics.invalidate_project_cache(project) └─→ Notification Log inserted (assignee / leader) └─→ after_insert → push_sender.send_push_for_notification └─→ filter by Vernon Push Preference └─→ POST to each Vernon Push Subscription endpoint └─→ dead endpoint (404/410) → prune row

4.2 Boot & offline read

PWA load → GET /m/ → www/m.py renders shell + injects CSP headers ↓ SW activate (cache name = "vt-{git_sha}") ↓ React mounts → GET /m/_boot → {user, roles, csrf_token, flags} ↓ useQuery(["my-work"]) → fetch /api/method/...my_work ├─ online → SW StaleWhileRevalidate: serve cache, fetch fresh, update cache └─ offline → SW returns cached response ↓ OfflineBanner shown; StaleBadge if >1h old
