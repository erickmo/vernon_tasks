8. Service Worker & Cache

8.1 Strategy

  • Workbox 7 generateSW via vite-plugin-pwa.
  • Precache app shell (HTML, JS, CSS, icons) — 5 entries.
  • Runtime: StaleWhileRevalidate on /api/method/vernon_tasks.*, max-age 1 day, max 50 entries.
  • Cache name: vt-{VITE_GIT_SHA}. Old caches purged on activate.
  • POST mutations bypass cache; fail with network error if offline.

8.2 Update flow

  1. Deploy → SW fetches new sw.js on next page load.
  2. install → precache new shell.
  3. activate → delete caches not matching vt-{new_sha}.
  4. Client reloads (no skipWaiting in current config; reload on next visit).

9. Auth & Session

9.1 Login flow

User → POST /api/method/login body: usr=<email>&pwd=<password> ← Set-Cookie: sid=...; HttpOnly; SameSite=Lax User → GET /m/_boot cookie: sid=... ← { user, roles, csrf_token, flags } (stored: user/roles → sessionStorage; csrf_token → memory) User → any mutation cookie + header X-Frappe-CSRF-Token

9.2 Session expiry

  • Any 401/403 from API → ReloginModal opens.
  • Current route + state preserved in modal state.
  • After successful relogin → modal closes, original mutation retried.

12. Observability

12.1 Telemetry pipeline

  • Client emits logEvent(event, props) → POST /api/method/...telemetry.log_event.
  • Server validates event name against allowlist; truncates props ≤ 2048 bytes; rate-limits 60/min/user.
  • Stores in Vernon Telemetry Event: user, event, props (JSON), ts.
  • Daily scheduler purge_old_telemetry deletes rows > 90 days.
  • Funnel analysis via Frappe Report Builder.

12.2 Required reports (pre-launch)

  1. PWA Boot Funnel — pwa_boot → login_success → page_view
  2. Error Rate — error_boundary count / page_view count
  3. Install Funnel — install_prompt_shown → install_accepted
  4. Mutation Health — task_complete count + failure rate
  5. Push Delivery — push_subscribed vs push_received

12.3 Server logs

  • Frappe error log captures stack traces; review daily during pilot.
  • nginx access logs for /m/* — monitor 4xx/5xx ratio.

13. Infrastructure & Deployment

13.1 Topology

  • Standard frappe-bench on Ubuntu 22.04+.
  • MariaDB 10.6+, Redis (queue + cache), nginx → gunicorn workers.
  • Node 18+ on build host (PWA build).

13.2 Deploy steps

cd ~/frappe-bench
git -C apps/vernon_tasks pull
cd apps/vernon_tasks/pwa && npm ci && npm run build
cd ~/frappe-bench
bench --site <site> migrate          # apply schema changes, run patches
bench --site <site> clear-cache
bench restart                          # gunicorn + workers

Service worker auto-busts via git-SHA cache name.

13.3 VAPID keys (one-time)

bench --site <site> execute vernon_tasks.task.api.push.cli_generate_vapid
# stores public + private keys in VT Settings

13.4 Rollback

  1. git checkout <prev SHA> in apps/vernon_tasks.
  2. Rebuild PWA, bench restart.
  3. SW activates old cache namespace; users get prior code.
  4. Schema migrations: forward-fix in next patch; never rollback migrations destructively.
