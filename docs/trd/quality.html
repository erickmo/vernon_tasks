14. Testing Strategy

14.1 Backend

  • bench run-tests --app vernon_tasks
  • Unit: services (point_calculator, scheduling_engine, velocity, burndown, forecast, risk).
  • Integration: API endpoints with role checks & boundary inputs.
  • Coverage targets: services ≥ 80 %, security.py ≥ 80 %.

14.2 PWA

  • Unit/component: Vitest + happy-dom + @testing-library/react.
  • Hook tests: useUndoableMutation, useInstallPrompt, useMediaQuery.
  • e2e: Playwright. Env-gated: PWA_BASE_URL, PWA_TEST_USER, PWA_TEST_PASS.
  • Critical e2e flows:
    • login → /m/work renders
    • complete task with undo cancel
    • log progress modal
    • offline read (airplane mode)
    • install prompt after 2 completes
    • responsive: BottomNav <768 / TopNav ≥768
  • TypeScript: tsc --noEmit in CI.

14.3 CI gates

  1. npm run typecheck (PWA)
  2. npm test -- --run (Vitest)
  3. npm run build (verify bundle)
  4. bench run-tests --app vernon_tasks
  5. (Optional) Playwright if staging available

15. Coding Standards

15.1 Python

  • Layer separation: Controller (validation, db_set) · Service (logic) · Page Handler (auth + IO).
  • Function length ≤ 40 lines; class < 300 lines / 5 responsibilities.
  • Named constants module-level; no magic strings/numbers.
  • DI: services take settings/config as params.
  • ignore_permissions=True only in internal services / scheduler.
  • Email sends wrapped in try/except (silent skip).
  • Prefer frappe.get_all() / frappe.get_doc() over raw SQL.

15.2 TypeScript

  • Strict mode; no any.
  • API payload types in pwa/src/api/types.ts shared across modules.
  • Components ≤ 150 lines; extract hooks for logic.
  • Inline styles + CSS variables (no styled-components / Tailwind).
  • id-ID strings centralized in pwa/src/i18n.ts.
