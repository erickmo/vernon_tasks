Portal Reports P5 — Design Spec

Date: 2026-05-18 Parent PRD: docs/superpowers/specs/2026-05-17-desktop-portal-foundation-design.html Status: Draft — pending review Owner: Erick Mo

1. Scope

Desktop portal Reports domain — executive dashboard at /portal/reports surfacing org-wide and cross-project analytics for Managers and Leaders. Three tabs: OKR (org health + KPI trends), Sprints (cross-project velocity + forecast + risk), Team (leaderboard + workload + overdue). Mounts behind portal_reports_enabled flag. Aggregates across the existing exec_analytics.py, analytics.py, and ic_analytics.py services — no business logic duplication.

In scope (P5): - /portal/reports top-level route, link in portal nav (Manager + Leader only). - Feature gate: portal_reports_enabled (Check, default 0) in VT Settings. - OKR tab (Manager only): OKR roll-up per project, org-level health score, per-KPI trend line charts. - Sprints tab (Manager + Leader): cross-project sprint velocity comparison, per-project forecast cards, risk flag matrix. - Team tab (Manager + Leader): leaderboard, per-member completion rate, overdue analysis, workload distribution. - Permission enforcement: server-side role checks + client-side tab gating; Member redirect to 403. - Caching: 5-min TTL on heavy cross-project aggregations, cache key includes role.

Non-goals (P5, deferred): - Drill-down into individual tasks from the Team tab (P5.1). - Exportable PDF/CSV reports (P5.2). - Custom date range picker beyond preset periods (P5.2). - Scheduled email digests (P5.3). - Per-project OKR detail — use /portal/okr (P2). - Per-sprint burndown detail — use /portal/projects/:id/sprints/:sprint (P3.2).

2. Architecture

2.1 Backend — vernon_tasks/api/portal_reports.py

New file at vernon_tasks/api/portal_reports.py — sibling to vernon_tasks/api/sprints.py and projects.py. Thin aggregation layer: no business logic copied from underlying services. All computation remains in task/services/. This module's sole responsibility is: 1. Portal permission check (role + feature flag). 2. Cross-project fan-out where a single service call covers only one project. 3. 5-minute result cache keyed by (endpoint, role, params). 4. Serialisation to portal-safe response shapes.

Calling pattern — internal imports (not whitelisted RPC calls to self):

from vernon_tasks.task.services.okr_rollup_service import get_okr_rollup as _okr
from vernon_tasks.task.services.kpi_trend_service import get_kpi_trend as _kpi_trend, list_kpis as _list_kpis
from vernon_tasks.task.services.health_score_service import get_health_score as _health
from vernon_tasks.task.services.velocity_service import get_velocity_trend as _get_velocity_trend
from vernon_tasks.task.services.forecast_service import get_forecast as _get_forecast
from vernon_tasks.task.services.risk_evaluator import evaluate_risks as _evaluate_risks
from vernon_tasks.task.services.leaderboard_service import get_leaderboard as _lb

Permission constants:

_MANAGER_ROLES  = ("VT Manager", "System Manager")
_LEADER_ROLES   = ("VT Leader", "VT Manager", "System Manager")
_CACHE_TTL_SEC  = 300  # 5 minutes

Guards:

def _require_leader():
    """Allows VT Leader + VT Manager."""
    if not set(frappe.get_roles()) & set(_LEADER_ROLES):
        frappe.throw("Not authorized", frappe.PermissionError)

def _require_manager():
    """Allows VT Manager only."""
    if not set(frappe.get_roles()) & set(_MANAGER_ROLES):
        frappe.throw("Not authorized", frappe.PermissionError)

def _check_flag():
    enabled = frappe.db.get_single_value("VT Settings", "portal_reports_enabled")
    if not int(enabled or 0):
        frappe.throw("Portal Reports is not enabled", frappe.PermissionError)

Each whitelisted endpoint calls _check_flag() first, then the appropriate role guard. The role is included in cache keys to prevent cross-permission leaks when a Manager and a Leader share a Frappe instance (different project scopes).

Cache helper (consistent with analytics.py pattern):

def _cache(key, fn):
    cached = frappe.cache().get_value(key)
    if cached is not None:
        return cached
    val = fn()
    frappe.cache().set_value(key, val, expires_in_sec=_CACHE_TTL_SEC)
    return val

ReportsFeatureGate.tsx implementation follows SprintsFeatureGate.tsx pattern — reads portal_reports_enabled via a deduplicated settings query, renders <ComingSoon domain="Reports" /> if 0, renders children if 1.

2.3 Feature flag

  • DocType: VT Settings
  • Field: portal_reports_enabled (Check, default 0, label "Enable Portal Reports")
  • Backend: _check_flag() called at top of every endpoint in portal_reports.py. Raises frappe.PermissionError (HTTP 403) when 0.
  • Frontend: ReportsFeatureGate queries portal_reports_enabled from the settings endpoint shared with other feature gates. Nav link in nav.ts is always registered (permission-gated); feature gate is a second layer inside the route.

useOkrReport.ts (hook):

export function useOkrReport(period?: string) {
  const health = useQuery({
    queryKey: ["reports", "health"],
    queryFn: () => getPortalHealthScore(),
    staleTime: 5 * 60 * 1000,
  });
  const rollup = useQuery({
    queryKey: ["reports", "okr", period ?? "current"],
    queryFn: () => getPortalOkrRollup(period),
    staleTime: 5 * 60 * 1000,
  });
  return { health, rollup };
}

4.3 React components

SprintsTab.tsx: - Parallel fetches via useSprintsReport hook. - n selector state (3 / 6 / 12), stored in component state. - Renders <VelocityComparisonChart>, <ForecastGrid>, <RiskMatrix>.

VelocityComparisonChart.tsx (recharts BarChart grouped): - X-axis: sprint label. Each project is a <Bar> series with distinct colour. - Legend lists project titles. - Tooltip: {sprint, project, velocity}.

useSprintsReport.ts (hook):

export function useSprintsReport(n: number) {
  const velocity = useQuery({
    queryKey: ["reports", "velocity", n],
    queryFn: () => getPortalVelocityComparison(n),
    staleTime: 5 * 60 * 1000,
  });
  const forecasts = useQuery({
    queryKey: ["reports", "forecasts"],
    queryFn: () => getPortalForecasts(),
    staleTime: 5 * 60 * 1000,
  });
  const risks = useQuery({
    queryKey: ["reports", "risks"],
    queryFn: () => getPortalRisks(),
    staleTime: 2 * 60 * 1000,   // shorter: risks more volatile
  });
  return { velocity, forecasts, risks };
}

6. Permissions

Enforced at two layers: 1. Server — role check in every endpoint. Returns HTTP 403 on violation. 2. Client — tab visibility gated by usePermissions().roles. Client check is UI hint only; server is authoritative.

Access /portal/reports route yes yes no — redirect 403
OKR tab (visible in UI) yes no no
get_portal_health_score endpoint yes 403 403
get_portal_okr_rollup endpoint yes 403 403
get_portal_kpi_list endpoint yes 403 403
get_portal_kpi_trend endpoint yes 403 403

Tab rendering logic in ReportsPage.tsx:

const { roles } = usePermissions();
const isManager = roles.includes("VT Manager") || roles.includes("System Manager");
const isLeader  = roles.includes("VT Leader");

const tabs = [

7. Caching Strategy

All heavy read endpoints in portal_reports.py use frappe.cache().set_value with TTL 300s (5 min).

Endpoint Cache Key Pattern TTL
get_portal_health_score pr:health:manager 300s
get_portal_okr_rollup pr:okr:{period}:manager 300s
get_portal_kpi_list pr:kpis:manager 300s
get_portal_kpi_trend pr:kpi_trend:{kpi}:{periods}:manager 300s
get_portal_velocity_comparison pr:vel:{role}:{n}:{user} 300s
get_portal_forecasts pr:forecasts:{role}:{user} 300s
get_portal_risks (no cache — risks are near-realtime) —
get_portal_leaderboard pr:leaderboard:{period}:{limit}:{role}:{user} 300s
get_portal_workload pr:workload:{role}:{user} 300s
get_portal_overdue pr:overdue:{role}:{user} 300s

Key design rules: - Manager endpoints use fixed suffix manager — no user suffix needed since Managers see all data identically. - Leader endpoints include {user} — each Leader may see a different project subset. - Role bucket (manager vs leader) is always in the key to prevent Manager-scoped cache being served to a Leader who later gains Manager role mid-session.

Cache invalidation: - Health score and OKR roll-up: invalidated by on_update hook on VT KPI Snapshot and VT OKR Period docs — add portal_reports.invalidate_okr_cache as a hook in hooks.py. - Velocity / forecast: piggyback on existing analytics.invalidate_project_cache hook — extend it to also call frappe.cache().delete_value(f"pr:vel:...") and pr:forecasts:... (role-bucket keys only, not user-keyed because iterating all users is expensive; instead, set TTL 300s and accept up to 5-min staleness). - Leaderboard / workload / overdue: no hook invalidation — rely on TTL only. Acceptable because team metrics are not time-critical at 5-min granularity.

Frontend staleTime: React Query staleTime: 5 * 60 * 1000 (5 min) mirrors backend TTL. Re-fetches triggered only on window focus after stale period, not on every navigation.

8. Testing

8.1 Backend — vernon_tasks/api/test_portal_reports.py

Test file at vernon_tasks/api/test_portal_reports.py. Follows pattern of test_sprints.py.

11. Open questions / risks

  1. Leader project scoping — project membership definition. Currently scoped to VT Project.assigned_to == frappe.session.user. If Leaders can be co-leaders of projects or lead sub-teams, this field is insufficient. Clarify before implementation: does a Leader relationship live on VT Project or on a separate assignment DocType?

  2. Velocity comparison chart readability. With 20+ projects each having 12 sprints, a grouped BarChart becomes unreadable. Mitigation: cap at top-N projects by avg velocity (configurable, default 10). UI shows "Showing top 10 by velocity" label. Deferred configurability to P5.2.

  3. Recharts bundle size. Recharts is ~320KB minified. Lazy loading per chart component mitigates initial load. Verify portal chunk budget does not bloat beyond 350KB gzip with recharts added. If it does, consider replacing CompletionRingChart (single metric) with an inline SVG arc (same technique as BurndownChart.tsx) to save the RadialBarChart import.

  4. _get_velocity_trend per-project fan-out latency. If a Manager oversees 50 projects, get_portal_velocity_comparison makes 50 service calls. Each call hits DB. Even at 20ms/call, that is 1 second before caching. For the first request after cache miss, this is potentially slow. Mitigation options: (a) lower fan-out cap to 20 projects; (b) rewrite velocity service to accept a list of projects in a single SQL pass. Track via reports.tab_view latency in telemetry.

  5. get_portal_risks no cache. Risk evaluation calls _evaluate_risks(project) per project with no caching. If risk evaluation is expensive (joins on tasks + sprints), this will be slow. Evaluate _evaluate_risks complexity before shipping; add 60s TTL if needed.

  6. Health score formula alignment. The formula score = okr_pct × 0.40 + ontime_pct × 0.30 + velocity_health × 0.30 is defined in health_score_service.py. portal_reports.py does not recompute it. If the formula changes, the breakdown labels in HealthScoreCard (OKR 40% / Ontime 30% / Velocity 30%) must be updated too. Consider serving the weights from the API response (they are included in get_portal_health_score → components field) and rendering them dynamically in the card.
