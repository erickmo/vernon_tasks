VT Item — Unified Hierarchy — Design Spec

Date: 2026-06-07 · Status: approved (brainstorm) · Scope: XL (collapses 5 doctypes into one nested-set tree, fresh-start data wipe, rewrites ~60 consumer files, phased P1–P4)

1. Goal

Collapse the strategy-to-execution hierarchy into a single navigable tree. Today five separate doctypes hold the chain:

Objective ──< Key Result
   │   └──< KPI Definition ──< KPI Entry
   └──< VT Project ──< VT Sprint ──< VT Task (self-link via parent_task)

After this change there is one doctype VT Item, a Frappe nested-set tree discriminated by node_type. The canonical order is:

OKR / KPI  →  Project  →  Sprint  →  Task

Two drivers, both in scope: (1) navigation/UX — one tree screen to drill OKR → Project → Sprint → Task; (2) data-model unification — literally one doctype, replacing the five.

3. VT Item doctype

Frappe tree doctype: "is_tree": 1, nested-set columns (lft, rgt, old_parent), parent field parent_vt_item, flag is_group. Controller extends frappe.utils.nestedset.NestedSet.

3.1 Field groups (fat node, depends_on: node_type)

GroupFields
Shared (all)node_type (Select: OKR/KPI/Project/Sprint/Task), title, parent_vt_item, is_group, brand (Link VT Brand), owner_user (Link User), description, percent_done, start_date, end_date

Sprintgoal, actual_velocity, burndown_actual_json, outcome, sprint_state

Status enums differ per type and are kept as separate depends_on fields rather than one merged enum: health_status (OKR/Project), sprint_state (Sprint), kanban_status (Task), pdca_phase (OKR/Project/Task variants). Task subtasks reuse the tree parent (parent_task retired — a subtask is simply a Task whose parent is another Task).

3.2 Child tables retained

  • Key Result — under OKR node (metric, target_value, current_value, unit, progress_percent, confidence).
  • KPI Entry — under KPI node (date, value, notes). Becomes a child table of VT Item instead of a standalone doctype.
  • Existing children stay: Project Team Member, Project Milestone, Project Documentation, Task Dependency, Task Schedule Entry.
  • Sprint Task child table is dropped — tasks are tree children of the sprint node, not table rows.

4. Hierarchy validation (strict + skips)

Controller validate() enforces legal parent type and rejects illegal nesting (e.g. a Project under a Task).

OKRroot / nonetop tier
KPIroot or OKRboth allowed
ProjectOKR or rootOKR optional (standalone OK)
SprintProject—
TaskSprint or ProjectSprint optional (backlog)

brand inherits from the nearest ancestor with a brand when left blank.

5. Rollup

On save, percent_done rolls up the chain Task → Sprint → Project → OKR as the mean of direct children's percent_done. Light derived metric, not the primary driver, but natural with nested set.

6. Tree UI

8. Fresh start (no migration)

Patch drop_legacy_hierarchy_doctypes deletes Objective, KPI Definition, KPI Entry (standalone), Key Result (standalone), VT Project, VT Sprint, VT Task, and the redundant Sprint Task child. Create VT Item + retained child tables. Rewrite demo_data.py to seed a node tree. Reload fixtures. No data is preserved — confirmed acceptable (dev/demo only).

9. Testing

  • Autoname per type produces the right prefix.
  • Parent validation: legal combinations pass; illegal (Project under Task, Sprint under OKR, etc.) raises.
  • Skip levels: Task with no Sprint (parent = Project) passes; Project with no OKR (parent = root) passes; KPI at root and under OKR both pass.
  • Rollup: child percent_done changes propagate to ancestors.
  • Brand inheritance: blank brand resolves from nearest ancestor.
  • Child tables: Key Result on OKR, KPI Entry on KPI persist and read back.
  • Smoke each rewired API via bench execute after its phase.
  • Tag each test file/function with the PRD/ADR id it covers.
