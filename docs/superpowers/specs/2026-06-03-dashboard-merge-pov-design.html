Dashboard Merge — POV Home — Design Spec

Date: 2026-06-03 · Status: approved (brainstorm) · Scope: M→L (consolidates 3 pages, re-scopes team data, deletes 2 pages, touches ~10 files)

1. Goal

Collapse the three coexisting desk dashboards into a single post-login home (vt-home) that always renders from the logged-in user's point of view:

  • My work — tasks assigned to me, my progress.
  • Projects I lead and projects I'm involved in — already on vt-home.
  • The team inside the projects I lead — currently lives on a separate, system-global leader dashboard; this is the core re-scope.

After this change there is one dashboard. my-dashboard and leader-dashboard are deleted.

2. Current state (as mapped)

Page Route Roles Nature Data source
vt-home ("Beranda") /app/vt-home Member/Leader/Manager Post-login landing, JS-only task.api.dashboard.me_progress / my_projects / schedule_agenda + onboarding.*
my-dashboard /app/my-dashboard Member/Leader/Manager Personal analytics page methods get_employee_stats / get_daily_completions / get_hours_summary / get_sprint_kanban (orphan)
leader-dashboard /app/leader-dashboard Leader/Manager System-global team aggregate page methods get_leader_stats / get_phase_distribution / get_team_leaderboard / get_overdue_tasks

Key gaps / bugs found during mapping:

  • leader-dashboard is global across the whole system, not scoped to the projects a user leads — contradicts the "my POV" intent.
  • my-dashboard hours summary is labeled "Hours" but the values and tooltip are in minutes (unit bug).
  • my-dashboard get_sprint_kanban is built but never rendered (orphan) — drop it.
  • Role model is decoupled: holding the VT Leader desk role does not make you the leader of any project, and being a project's project_leader does not grant the role. Dashboard "led" bucket keys solely on the project_leader Link field (_user_project_ids() in dashboard.py).

3. Locked decisions

  1. Team scope: Leader → only projects they lead; Manager/admin → global.
  2. Old pages: delete my-dashboard + leader-dashboard entirely (fold into vt-home).
  3. Layout: tabs — Beranda (always) and Tim (eligible users only).
  4. Analytics: keep daily-completions chart and hours summary; fix the hours unit bug.
  5. Tim tab trigger: data-driven — visible to anyone who leads ≥ 1 project (led-scope), plus VT Manager / admin (global-scope). The VT Leader desk role alone is no longer required; backend role guard is loosened accordingly.

4. Target architecture

vt-home stays the single dashboard Page (/app/vt-home, roles Member/Leader/Manager, post-login landing). vt_home.js gains a two-tab shell. All read-aggregation lives in task/api/dashboard.py (Vernon Frappe rule: dashboard reads belong in the API module, not page .py). The two page modules are deleted with their methods migrated/re-scoped into dashboard.py.

/app/vt-home  (Page, JS-only presentation)
├── Tab: Beranda  (always)            ← personal POV
│     me_progress · my_projects · schedule_agenda · onboarding.*
│     + personal_stats · daily_completions · hours_summary   (folded from my-dashboard)
└── Tab: Tim      (if team_tab_state.visible) ← leadership POV
      team_overview()  [scope resolved server-side]   (folded + re-scoped from leader-dashboard)

5. Tab 1 — Beranda (personal, all roles)

Existing vt-home spine is unchanged:

  • Hero greeting + "Buat Proyek" action, onboarding checklist
  • Workload cards (Task Terbuka / Terlambat / Jatuh Tempo) — me_progress.workload
  • Progres Saya: velocity chart, active sprint card, next-actions — me_progress
  • Proyek Saya: led cards + "Sebagai Anggota" list — my_projects
  • Jadwal 8 Hari — schedule_agenda, Akses Cepat quick links

New (folded from my-dashboard), placed inside "Progres Saya":

  • Daily completions bar chart — daily_completions (self-scoped).
  • Hours summary — hours_summary (self-scoped). Bug fix: convert minute sums to hours (÷ 60, 1 decimal), label "Jam", split Tercatat (logged) vs Sisa (remaining); tooltip in hours.
  • get_sprint_kanban (orphan) is not migrated.

6. Tab 2 — Tim (leaders, re-scoped)

Folded from leader-dashboard, rendered only when team_tab_state.visible:

  • KPI cards: Pending Review, Approval Rate %, Team Points (Bulan)
  • PDCA Phase Distribution donut
  • Team Points Leaderboard (top 10)
  • Overdue Tasks table
  • A link to leader-review for approve/reject actions (that page stays).

Scope rules (resolved server-side):

Caller scope Task filter applied to all 4 blocks
Admin (System Manager / Vernon Admin) global none
VT Manager global none
Leads ≥ 1 project (else) led task.project IN (led_project_ids)
None of the above — not eligible → no tab, team_overview raises PermissionError

led_project_ids comes from the existing _user_project_ids(user).led (projects where project_leader == user). Definitions preserved from the old methods: Pending Review = kanban_status='In Review' AND pdca_phase='CHECK'; Approval Rate = this-month DONE with revision_count=0 / this-month DONE; Team Points = sum earned_points of this-month DONE.

7. Data layer

All in task/api/dashboard.py, each begins with require_login().

# Personal (self-scoped to frappe.session.user) — migrated from my_dashboard page
personal_stats()        -> {...}                  # from get_employee_stats
daily_completions()     -> [{date, count}]        # from get_daily_completions
hours_summary()         -> {logged_hours, remaining_hours}   # from get_hours_summary, minutes→hours

# Team — migrated + re-scoped from leader_dashboard page
team_tab_state()        -> {visible: bool, scope: "led"|"global", led_count: int}   # cheap; drives tab visibility
team_overview()         -> {scope, stats, phase_distribution, leaderboard, overdue} # heavy; lazy on Tim click
  • team_tab_state() is cheap (a role check + one count) and is safe for everyone; it never throws — returns visible:false for the ineligible.
  • team_overview() re-resolves scope via a private _team_scope(user) and raises PermissionError if not eligible (defense-in-depth, mirrors old per-method guards).
  • _team_scope(user): global if admin or VT Manager; else led if _user_project_ids(user).led non-empty; else raise.
  • The four team blocks share one led_ids filter helper so the led/global branch is written once (no duplicated SQL).
  • Reuse existing constants/helpers (_user_project_ids, _is_admin, PDCA maps). Watch the known PRIORITY_RANK (no Critical) quirk — out of scope to fix here, but do not regress.

8. Frontend (vt_home.js)

  • Add a tab bar (two buttons: Beranda / Tim) toggling two content containers. Keep the IIFE wrapper and the existing presentation-only style (functions < 40 lines, named constants).
  • On load: render Beranda; call team_tab_state() — if visible, show the Tim tab button.
  • Tim data is lazy — team_overview() fires on first Tim click; reuse chart instances on refresh (donut/bar .update()), matching the old pattern to avoid duplicate canvases.
  • New named constants for the folded charts (heights, colors) alongside existing VELOCITY_CHART_HEIGHT / BRAND_BLUE.
  • UI strings in Bahasa Indonesia.

10. Testing

  • test_vt_home.py — keep page-exists + role-gating; add: tab shell renders; Tim tab hidden for a Member with 0 led projects, shown for a project leader.
  • dashboard.py API tests:
  • personal_stats / daily_completions / hours_summary self-scope; assert hours unit (minutes ÷ 60).
  • team_tab_state: member-no-led → visible:false; leads ≥1 project → visible:true, scope:"led"; Manager → visible:true, scope:"global".

  • Migrate the meaningful assertions from test_my_dashboard / test_leader_dashboard (metric definitions: approval rate, pending-review filter, points) into the new API tests before deleting.
  • Test-data note: VT Project requires brand + project_owner (MandatoryError otherwise — known fixture debt); fixtures must set both.

11. Out of scope (unchanged)

my_work, leader_review, my_analytics, leader_analytics, exec_analytics, vt_scorecard. The PRIORITY_RANK (missing Critical) and other pre-existing dashboard.py quirks are not fixed here beyond not regressing them.

12. Risks

  • Tab visibility flicker: Tim tab appears after team_tab_state() resolves; render the tab bar first, reveal the Tim button on callback to avoid layout jump.
  • Scope leakage: team_overview must re-resolve scope server-side (never trust a client-supplied scope) — team_tab_state is a hint only.
  • Nav drift: missing one of the §9.3 links leaves a 404; grep my-dashboard / leader-dashboard across .js, fixtures, and navbar_items before declaring done.
