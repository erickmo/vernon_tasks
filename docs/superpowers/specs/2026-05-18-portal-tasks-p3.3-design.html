Portal Tasks P3.3 — Design Spec

Date: 2026-05-18 Parent PRD: docs/superpowers/specs/2026-05-18-portal-sprints-p3.2-design.md Status: Draft — pending review Owner: Erick Mo

1. Scope

Desktop portal Task domain — task detail, task creation, and activity/comments inside the sprint board. Mounts within the existing pwa/src/portal/sprints/ module as an overlay/modal layer; no new top-level route. Closes the gap where P3.2 delivered a read+drag board but no task CRUD or detail access from the portal.

In scope (P3.3): - Task detail panel (slide-over): clicking any task card in TaskBoard opens a right-side slide-over panel showing full task fields and edit controls (permission-gated). - Task create modal: + button on TaskBoard opens a create modal; new task auto-linked to current sprint + project. - Comments and activity log: threaded comments (Frappe Comment doctype) + field-change history (Frappe Version doctype) rendered inside the detail panel. - Sprint Task child table hard removal — P3.2 soft-deprecated it; P3.3 drops it after verifying zero new writes via telemetry (see §9).

Non-goals (P3.3, deferred): - Inline edit directly on the kanban card (no double-click-to-edit). - Delete task from portal. - Cross-sprint task drag. - Velocity history / sprint completion trend chart. - Bulk task operations from portal. - Attachment upload. - @mention autocomplete in comments.

2. Architecture

2.1 Backend module — vernon_tasks/api/portal_tasks.py

Sibling to vernon_tasks/api/sprints.py. All functions are @frappe.whitelist(), no classes. Exposed as vernon_tasks.api.portal_tasks.*.

Functions: - get_task_detail(task) — full task doc + permitted edit fields list. - update_task(task, payload) — write mutable fields; server enforces perm matrix. - create_task(payload) — insert new VT Task; auto-sets sprint, project, creation. - get_task_comments(task) — returns comments (Frappe Comment) + version history entries, merged and sorted by creation ASC. - add_comment(task, content) — inserts Frappe Comment doc. - delete_comment(comment_name) — validates that comment.owner == frappe.session.user or user is Manager/Leader; hard-deletes.

2.2 Schema delta

No new DocType fields required. All fields used (title, deadline, assigned_to, kanban_status, priority, base_points, pdca_phase, completion_date, project, sprint, estimated_hours, kanban_rank) already exist on VT Task (P3.2 added kanban_rank).

Sprint Task child table removal: - Confirm zero writes for 2+ closed sprints via tasks.sprint_task_write telemetry counter (set up in P3.2 move_task). - If confirmed: remove child table field from VT Sprint DocType JSON via patch, drop tabSprint Task in a subsequent migration. - This is gated on observation window, not blocked on P3.3 feature delivery.

2.3 Frontend module — pwa/src/portal/tasks/

New directory, consumed by components in pwa/src/portal/sprints/ via relative import.

pwa/src/portal/tasks/
  TaskDetailPanel.tsx       // slide-over container + edit form
  TaskCreateModal.tsx       // modal for new task
  ActivityLog.tsx           // merged comment + version history list
  CommentThread.tsx         // comment composer + per-comment render
  api/
    tasks.ts                // RPC wrappers for portal_tasks.*
    types.ts                // TaskDetail, Comment, ActivityEntry, CreateTaskPayload, UpdateTaskPayload
  hooks/

Integration into existing sprint module: - TaskCard.tsx gains an onClick prop — calls onTaskOpen(task.name) passed down from TaskBoard. - TaskBoard.tsx owns selectedTask: string | null state, renders <TaskDetailPanel taskName={selectedTask} onClose={() => setSelectedTask(null)} />. - TaskBoard.tsx gains a + button (Manager/Leader only in toolbar; Member sees it only if sprint is Active) that opens <TaskCreateModal sprint={sprintId} project={projectId} onCreated={...} />.

3. Task Detail Panel

3.1 Slide-over behavior

  • Triggered by clicking any TaskCard in TaskBoard.
  • Renders as a fixed right-side drawer: position: fixed; top: 0; right: 0; width: 480px; height: 100vh. Overlays the board without unmounting it.
  • Escape key and overlay-click outside the panel both close it; onClose clears selectedTask.
  • Panel is a single React subtree; does not affect the board's DnD context.
  • Only one task can be open at a time; clicking a second card replaces the current panel (no query re-use, new fetch).

3.2 Data loading

useTaskDetail(taskName) calls get_task_detail with React Query key ["taskDetail", taskName]. staleTime: 30_000. On panel open, if the task already exists in ["sprintDetail", sprintId] cache, it is used as placeholderData to avoid blank flash; the full detail fetch runs in background.

get_task_detail returns:

{
  "task": {
    "name": "VT-TASK-00042",
    "title": "...",
    "deadline": "2026-05-25",
    "assigned_to": "mo@intinusa.id",
    "assigned_to_full_name": "Erick Mo",
    "kanban_status": "In Progress",
    "priority": "High",
    "base_points": 5.0,
    "pdca_phase": "DO",
    "completion_date": null,
    "project": "VT-PROJ-00001",

permitted_fields is computed server-side per the perm matrix (§6). The frontend renders each field as editable only if it appears in this list. Fields absent from the list render read-only.

Fields with edit permission render as controlled inputs; blur or explicit Save triggers update_task. Single-field saves (no bulk submit button) follow the existing portal OKR KR-update pattern. Each save shows an inline spinner on the field row and a success/error toast.

title uses a borderless <input type="text"> that becomes editable on click (pencil icon on hover). Required; empty string is a validation error displayed inline.

kanban_status, priority, pdca_phase use <select> dropdowns with the same value sets as KanbanStatus, Priority, and PdcaPhase types from pwa/src/portal/sprints/api/types.ts.

assigned_to is a <select> populated from project members (reuses the project member list already fetched in SprintDetail's project_summary). If the user is a Member, their own email is the only option shown (can only re-assign to themselves, which is a no-op; field is effectively read-only for Member).

deadline and estimated_hours use <input type="date"> and <input type="number" step="0.5" min="0"> respectively.

sprint, project, base_points, completion_date are always read-only display rows.

3.4 Optimistic updates from the panel

On save of any field: 1. Optimistically update ["taskDetail", taskName] cache. 2. Optimistically update the matching task entry in ["sprintDetail", sprintId] cache so the kanban card reflects the change immediately. 3. POST update_task. On failure: rollback both caches, show error toast quoting server message.

4. Task Create Flow

4.1 Trigger

+ button rendered in TaskBoard's column toolbar or in a fixed position at the top of the board. Visibility: Managers and Leaders always see it. Members see it only when sprint.status === "Active" (controlled by canCreate flag derived from session role + sprint status, evaluated client-side as a hint, enforced server-side).

4.2 Modal fields

TaskCreateModal is a centered dialog (not a slide-over). Fields:

Field Input Required Default
title text yes —
priority select (Low/Medium/High/Critical) yes Medium
estimated_hours number, step 0.5, min 0 no 1.0
deadline date no —
assigned_to select (project members) no session.user
pdca_phase select (BACKLOG/PLAN/DO/CHECK/ACT/DONE) no BACKLOG
kanban_status select (all 7 values) no Backlog

Hidden / auto-set: sprint = sprintId, project = projectId. Not shown in modal.

Validation (client-side, mirrored server-side): - title non-empty after trim; max 140 chars. - estimated_hours ≥ 0. - deadline ≥ today if set (warning, not hard block).

4.3 API — create_task

@frappe.whitelist()
def create_task(payload):
    payload = _parse_payload(payload)
    # Permission: Member only if sprint.status == "Active"
    sprint = frappe.get_doc("VT Sprint", payload["sprint"])
    role = _get_user_role(payload["project"])
    if role == "Member" and sprint.status != "Active":
        raise frappe.PermissionError("Members can only create tasks in Active sprints")
    doc = frappe.get_doc({
        "doctype": "VT Task",
        "title": payload["title"],
        "sprint": payload["sprint"],
        "project": payload["project"],
        "priority": payload.get("priority", "Medium"),
        "estimated_hours": payload.get("estimated_hours", 1.0),
        "deadline": payload.get("deadline"),
        "assigned_to": payload.get("assigned_to", frappe.session.user),
        "pdca_phase": payload.get("pdca_phase", "BACKLOG"),
        "kanban_status": payload.get("kanban_status", "Backlog"),
        "kanban_rank": None,  # lazy-populated on next board load
    }).insert()
    return {"name": doc.name, "task": frappe.db.get_value(
        "VT Task", doc.name,
        ["name", "title", "assigned_to", "kanban_status", "pdca_phase",
         "kanban_rank", "estimated_hours", "weight", "priority", "deadline"],
        as_dict=True,
    )}

Response shape matches TaskCardData so it can be directly inserted into the board cache.

4.4 Optimistic insert

On submit: 1. Generate a temporary ID tmp-<uuid> and insert a provisional TaskCardData into the relevant column of ["sprintDetail", sprintId] cache. Temporary card renders with a task-card--pending class (slight opacity, no drag handle). 2. POST create_task. 3. On success: replace the provisional entry with the real server response (swapping name: tmp-<uuid> for the real doc name). 4. On failure: remove provisional entry, show error toast. 5. Modal closes on success; on failure modal stays open with the error displayed inline below the submit button.

4.5 Rank assignment for new task

kanban_rank is returned as null from create_task. The optimistic insert places the card at the bottom of its column (rank = max existing rank + 1000, computed client-side). The server will populate via the existing _lazy_populate_ranks path on the next get_sprint_with_relations call.

5. Comments and Activity Log

5.1 Data model

Two separate sources merged into a single timeline:

Comments — Frappe native Comment doctype: - reference_doctype = "VT Task", reference_name = task.name - comment_type = "Comment" (user-authored), comment_type = "Info" (system notes, e.g. assignment changes) - Fields used: name, owner, creation, content (HTML from Frappe rich-text editor)

Version history — Frappe native Version doctype: - ref_doctype = "VT Task", docname = task.name - Fields used: name, owner, creation, data (JSON string with changed array: [[field, old_value, new_value], ...]) - Only surface changes to a curated field set: kanban_status, pdca_phase, priority, assigned_to, deadline, estimated_hours.

Merge: get_task_comments returns a unified ActivityEntry[] sorted by creation ASC. Each entry has a type discriminant: "comment" or "version".

5.2 API — get_task_comments

@frappe.whitelist()
def get_task_comments(task):
    _assert_task_readable(task)

    comments = frappe.get_all(
        "Comment",
        filters={"reference_doctype": "VT Task", "reference_name": task,
                 "comment_type": ["in", ["Comment", "Info"]]},
        fields=["name", "owner", "creation", "content", "comment_type"],
        order_by="creation asc",
        limit=200,
    )

    TRACKED_FIELDS = {"kanban_status", "pdca_phase", "priority", "assigned_to", "deadline", "estimated_hours"}
    versions = frappe.get_all(
        "Version",
        filters={"ref_doctype": "VT Task", "docname": task},
        fields=["name", "owner", "creation", "data"],
        order_by="creation asc",
        limit=200,
    )
    version_entries = []
    for v in versions:
        try:
            changes = json.loads(v["data"]).get("changed", [])
        except Exception:
            continue
        filtered = [[f, o, n] for f, o, n in changes if f in TRACKED_FIELDS]
        if filtered:
            version_entries.append({
                "name": v["name"],
                "owner": v["owner"],
                "creation": str(v["creation"]),
                "type": "version",
                "changes": filtered,
            })

    comment_entries = [
        {**c, "type": "comment", "creation": str(c["creation"])}
        for c in comments
    ]

    merged = sorted(comment_entries + version_entries, key=lambda e: e["creation"])
    return merged

5.3 API — add_comment

@frappe.whitelist()
def add_comment(task, content):
    _assert_task_readable(task)
    if not content or not content.strip():
        frappe.throw("Comment content is required")
    doc = frappe.get_doc({
        "doctype": "Comment",
        "comment_type": "Comment",
        "reference_doctype": "VT Task",
        "reference_name": task,
        "content": content,
    }).insert(ignore_permissions=True)
    return {
        "name": doc.name,
        "owner": doc.owner,
        "creation": str(doc.creation),
        "content": doc.content,
        "type": "comment",
        "comment_type": "Comment",
    }

5.4 API — delete_comment

@frappe.whitelist()
def delete_comment(comment_name):
    doc = frappe.get_doc("Comment", comment_name)
    role = _get_user_role_for_task(doc.reference_name)
    if role not in ("Manager", "Leader") and doc.owner != frappe.session.user:
        raise frappe.PermissionError("Cannot delete another user's comment")
    frappe.delete_doc("Comment", comment_name, ignore_permissions=True)
    return {"ok": True}

5.5 Frontend rendering

ActivityLog.tsx — receives ActivityEntry[], renders a scrollable list. Two render variants: - type === "comment": renders the comment bubble (see CommentThread). - type === "version": renders a compact diff line per changed field, e.g. "kanban_status: Backlog → In Progress". Uses a FIELD_LABELS map for human-readable names.

CommentThread.tsx — handles the comment composer + per-comment display: - Composer: <textarea> (auto-resize, max 1000 chars), Submit on Ctrl+Enter or explicit Send button. Clears after submit. - Per comment: avatar initial + owner, relative time ("2 min ago" → "May 18" after 24h), content rendered as sanitized HTML (using DOMPurify — already a dependency; if not, add it). Delete button (trash icon) shown only to comment owner and Manager/Leader. - Optimistic add: insert a provisional entry with name: tmp-<uuid>, type: "comment", pending CSS class. On success replace; on failure remove + toast. - Optimistic delete: remove from local list immediately, rollback on server error.

useTaskComments(taskName) — React Query key ["taskComments", taskName]. staleTime: 10_000. Exposes { entries, addComment, deleteComment }.

Activity panel is rendered in the lower half of TaskDetailPanel with a fixed-height scroll container (flex: 1; overflow-y: auto). New comments appear at the bottom; panel auto-scrolls after successful add.

6. Permissions

Permission matrix enforced server-side in every portal_tasks.py function. Client-side enforcement is a hint only (UI shows/hides controls), never relied upon for security.

_get_user_role(project) reuses the same helper pattern as sprints.py — checks VT Project Member table for the session user's role in the given project.

Action Manager Leader (own project) Member
View task detail yes yes yes (where project visible)
Edit title yes yes own task only
Edit kanban_status yes yes own task only
Edit pdca_phase yes yes own task only
Edit priority yes yes no
Edit estimated_hours yes yes no
Edit assigned_to yes yes no
Edit deadline yes yes no
Create task in sprint yes yes Active sprint only
Add comment yes yes yes (any visible task)
Delete own comment yes yes yes
Delete others' comment yes yes no

permitted_fields is computed as:

def _permitted_fields(task_doc, project, role):
    always_editable = []
    if role in ("Manager", "Leader"):
        return ["title", "deadline", "assigned_to", "kanban_status", "priority",
                "estimated_hours", "pdca_phase"]
    if role == "Member" and task_doc.assigned_to == frappe.session.user:
        return ["title", "kanban_status", "pdca_phase"]
    return []  # read-only for Members on others' tasks

10. Open Questions / Risks

  • DOMPurify dependency: confirm dompurify + @types/dompurify is already in pwa/package.json. If not, add before P3.3 implementation. Rendering raw HTML from Frappe's Comment content without sanitization is a stored XSS risk.
  • frappe.DoesNotExistError vs frappe.ValidationError: existing sprints.py already uses raise frappe.DoesNotExistError(...) in two places (lines 74, 136) despite P3.2 spec noting this is not a valid exception. The implementation diverged from spec. P3.3 portal_tasks.py must NOT follow this pattern — use frappe.throw(...) (raises frappe.ValidationError) for not-found cases, consistent with the spec constraint.
  • Comment HTML sanitization on backend: Frappe stores rich-text comment content as HTML. Consider stripping on read in get_task_comments using frappe.utils.strip_html_tags if a plain-text fallback is preferred, or documenting that front-end sanitization via DOMPurify is the sole safeguard.
  • assigned_to user list source: get_task_detail must return the project member list for the assigned_to dropdown. Current approach relies on project_summary already in the sprint cache. If the panel is opened from a context without that cache (future: standalone task link), a separate list_project_members(project) call may be needed. Defer to P3.3+ unless the standalone case is required.
  • Version track_changes coverage: relies on VT Task having track_changes = 1 in DocType JSON (verified true per P3.2 spec note). Confirm the six tracked fields (kanban_status, pdca_phase, priority, assigned_to, deadline, estimated_hours) are not excluded from the tracked set.
  • Concurrent edits: two users editing the same task simultaneously will produce last-write-wins behavior (standard Frappe doc.save() pattern). No optimistic locking implemented in P3.3. Add modified timestamp comparison in update_task as a simple conflict guard if needed.
