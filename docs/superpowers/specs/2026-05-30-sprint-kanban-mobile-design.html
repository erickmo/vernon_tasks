Sprint Kanban Mobile

Date: 2026-05-30
Scope: v2 — a mobile sprint board at /m/sprint/:sprintId showing a task kanban with horizontal-scroll columns, drag-to-move via @dnd-kit, and an axis toggle (kanban_status ↔ pdca_phase) mirroring the desktop board.
Approach (approved): Drag (@dnd-kit) for moving tasks; axis TOGGLE between both column sets.
Pre-state (verified): Backend move_task() + get_sprint_with_relations() + rebalance_column() already exist and are production-ready — NO new Python. @dnd-kit v6 + sortable v8 already installed and used by desktop portal/sprints/TaskBoard.tsx. Mobile has only a read-only SprintCommitmentCard on MeTab; no sprint board. KanbanCard/KanbanColumn stubs exist but are unused and not dnd-wired. Fractional-rank helper at portal/sprints/lib/rank.ts.

1. Architecture

FileResponsibility
mobile/pages/Sprint/api.tsThin wrappers: getSprintBoard(sprintId) → get_sprint_with_relations; moveTask(payload) → move_task; rebalanceColumn(...) → rebalance_column. Reuse the portal request client + types where possible.
mobile/pages/Sprint/hooks/useSprintBoard.tsreact-query: load board (tasks grouped by current axis), moveTask mutation with local reorder + invalidate, rebalance on rank collision (reuse lib/rank.ts).
mobile/pages/Sprint/SprintBoardMobile.tsxPage: header (sprint title, axis toggle), horizontal-scroll column strip, DndContext.
mobile/pages/Sprint/MobileKanbanColumn.tsxDroppable column: title + count + scrollable card list (uses @dnd-kit useDroppable / SortableContext).
mobile/pages/Sprint/MobileTaskCard.tsxDraggable card: title, assignee, priority dot, hours. (Adapt portal TaskCard look; mobile spacing.)

Reuse where clean: the portal api/types.ts (TaskCardData, KanbanStatus, PdcaPhase) and lib/rank.ts (computeRank, needsRebalance). Do NOT reuse the unused components/KanbanCard/KanbanColumn stubs (wrong shape; they link to /m/project) — build mobile-specific ones.

2. Columns & axis toggle

  • kanban_status (7): Backlog, Scheduled, In Progress, In Review, Revision, Done, Blocked.
  • pdca_phase (6): BACKLOG, PLAN, DO, CHECK, ACT, DONE.
  • Toggle in the header (segmented control) flips which field groups the columns. Default: kanban_status.
  • Columns render in a horizontal-scroll strip (each ~80vw min-width, snap scroll), since 6–7 columns can't fit a phone width.

3. Move flow (drag)

  • Library: @dnd-kit (already installed). DndContext with PointerSensor (small activation distance so taps still work) + TouchSensor with a short delay so vertical page scroll vs drag is distinguishable.
  • Within column: reorder via SortableContext; compute new kanban_rank with computeRank(prev, next).
  • Across columns: on drop into a different column, call moveTask({ task, [axisField]: targetColumn, kanban_rank }). Optimistically reorder local state, then fire mutation; on error, refetch (rollback).
  • Rank collision: if needsRebalance(rank), call rebalanceColumn(sprint, axis, columnValue) then refetch.
  • Permissions: server enforces VT Member restrictions (can't mark Done, etc.) and returns PermissionError; surface as a toast and rollback. Do NOT replicate permission logic client-side beyond disabling obviously-forbidden affordances if cheap.

4. Entry point

  • Make the mobile SprintCommitmentCard (MeTab) tappable → navigate to /m/sprint/:sprintId for the active sprint.
  • Register the route in the mobile router. Guard behind existing auth/permission guards used by other mobile pages.
  • Back navigation returns to the dashboard.

5. States

  • Loading → page skeleton (reuse PageSkeleton/Skeleton).
  • Empty sprint (no tasks) → EmptyState.
  • Error loading → EmptyState with retry.
  • Move in-flight → card shows a subtle pending style; on failure a toast + rollback.
  • Offline → moves disabled with a hint (this feature does NOT integrate the offline outbox; out of scope here to keep the two v2 streams independent).

6. Telemetry

Add to TelemetryEvent union: sprint_board_open ({sprint}), sprint_task_move ({from, to, axis}), sprint_axis_toggle ({axis}).

7. Testing

  • api.ts: wrappers call the right endpoints with the right params (mock request client).
  • useSprintBoard: groups tasks by axis; moveTask mutation calls API + invalidates; axis toggle regroups.
  • MobileTaskCard / MobileKanbanColumn: render title/assignee/priority; column shows count.
  • SprintBoardMobile: renders columns for the active axis; toggling axis swaps column set; a simulated drop calls moveTask with target column (use @dnd-kit testing approach or directly invoke the onDragEnd handler with a synthetic event).
  • Entry point: tapping the sprint card navigates to /m/sprint/:id.
  • Gate: full pnpm vitest run + tsc + build.

8. Out of scope

  • New backend endpoints (none needed).
  • Offline queuing of moves (separate v2 stream).
  • Creating/editing sprints from mobile (desktop portal handles it).
  • Tap-to-move fallback (drag only, per decision).
  • Burndown/velocity on this page (already on LeaderSprint).
