Project Detail UI/UX + Embedded Kanban

Date: 2026-05-31
Task size: M (new read + 3 mutation endpoints, page JS rewrite, new css; ~6 files)
App ownership: 100% vernon_tasks — no Frappe core files touched
Scope: Rewrite the vt-project-detail desk page into a hero + tabbed layout whose primary tab is an embedded, drag-driven Kanban board for managing the project's tasks.

1. Goal

Turn the read-only project detail dashboard into a project management surface. The user manages tasks without leaving the page: drag a card to change its status, quick-add a task per column, click a card to open its form, and inline-edit priority / assignee / deadline from the card.

2. Layout

  • Hero (kept): title, status chip, risk chip, progress bar, leader / dates / PDCA phase.
  • Tabs: Kanban (default) · Milestone · Tim · Ringkasan. Non-kanban tabs reuse the existing section renderers.

3. Domain constraint — PDCA state machine

kanban_status is derived from pdca_phase via PDCA_KANBAN_MAP in the VT Task controller. Blocked is an orthogonal flag. Legal PDCA transitions (Deming cycle):

BACKLOG → PLAN → DO → CHECK → {ACT, DONE, DO}
ACT → DO
DONE = terminal
Blocked = orthogonal (reachable from any state; un-block restores PDCA-derived status)

Therefore the board is not free drag-anywhere. Move mechanics (revised after review):

  • Drop on a PDCA column → clear Blocked flag first (unconditional — _sync_kanban_status is Blocked-sticky), set pdca_phase (reverse of PDCA_KANBAN_MAP), then doc.save() without ignore_validate so _validate_pdca_transition enforces legality. Illegal → throws → JS snaps the card back + toast.
  • Move into Done (legal only from In Review = CHECK): also stamp completion_date = today(). Required because VT Task is submittable and on_submit does NOT fire on a plain save() — mirrors my_work_mutations.complete, but keeps the transition guard.
  • Drop on Blocked → set kanban_status = KANBAN_BLOCKED directly (orthogonal flag), idempotent if already Blocked. pdca_phase untouched.
  • Drag out of Blocked → un-block is NOT a transition: restore kanban_status = PDCA_KANBAN_MAP[pdca_phase] (snap card to its real phase column) and ignore the dropped-on column.
  • DONE is terminal → Done cards are locked (not draggable, allowed_targets = []). Re-opening is out of scope.
  • Submitted tasks (docstatus == 1) → save() is blocked by Frappe; move_task detects and rejects with a friendly message (snap-back), never a 500. Stale-edit TimestampMismatchError → JS reverts + reloads the board.
  • On drag-start the board highlights legal target columns and grays illegal ones, using allowed_targets returned per task by the read API. Computed from the task's real pdca_phase via VALID_PDCA_TRANSITIONS; a Blocked card's only target is its real-phase column (un-block); every non-Done card also gets Blocked as a target.

4. API surface

Read — dashboard.project_board(project_id)

Whitelisted. Guarded by existing _assert_project_access. Returns all project tasks grouped into the 7 columns, plus the team roster for the assignee dropdown:

{
  "columns": [
    {"key": "Backlog", "label": "Backlog", "pdca_phase": "BACKLOG",
     "tasks": [{"id","title","priority","deadline","assigned_to","risk_flag",
                "kanban_status","allowed_targets": ["Scheduled","Blocked"]}]},
    ... (Scheduled, In Progress, In Review, Revision, Done, Blocked)
  ],
  "team": [{"user": "...", "full_name": "..."}],
  "priorities": ["Low","Medium","High","Critical"]
}

Within a column, tasks are ordered by priority (Critical→Low) then deadline asc. allowed_targets computed per task from its real pdca_phase via imported VALID_PDCA_TRANSITIONS mapped to column labels (+ Blocked for non-Done cards; Blocked card → its real-phase column only; Done → empty). To avoid duplicating the _detail_open_tasks query, extract a shared _project_tasks(project_id, *, include_closed) fetcher + row mapper; both _detail_open_tasks and project_board call it. Tasks whose kanban_status is none of the 7 known columns (e.g. legacy Cancelled) are skipped defensively.

Mutations — new module task/api/board_mutations.py

Three thin whitelist wrappers (≤10 lines each) that delegate to the VT Task controller via doc.save() / doc.insert() so the PDCA state machine does the enforcing. All call rate_limit first. Authz is project-level: each asserts caller has project access (admin / leader / member of the task's project) via security.assert_project_access (lifted out of dashboard.py), then saves with ignore_permissions=True. Constants (PDCA_KANBAN_MAP, VALID_PDCA_TRANSITIONS, KANBAN_BLOCKED, BOARD_COLUMNS) are imported from the controller — never re-declared.

FunctionBehavior
move_task(task_id, to_column)rate_limit("move_task", 60). Reject if docstatus==1. Blocked target → set kanban_status flag (idempotent). Un-block (Blocked→PDCA) → restore kanban_status from pdca_phase, no transition. PDCA target → clear Blocked, set pdca_phase from reverse map, stamp completion_date if Done, save() (no ignore_validate). Returns new kanban_status or throws.
create_task(project_id, title, column)rate_limit("create_task", 30). Explicit fields only (no mass-assign): project, max_str(title), pdca_phase from column. Quick-add allowed only on PDCA columns (not Blocked/Done). weight defaults to 1.
patch_task(task_id, field, value)rate_limit("patch_task", 30). Allowed fields whitelist: priority, assigned_to, deadline. Reject anything else. assigned_to must be a project team member. Set + save.

5. Frontend

  • vt_project_detail.js rewritten: hero + tab bar + tab panels. Kanban panel built with Sortable.js (bundled with Frappe — verify window.Sortable at plan step).
  • Each column is a Sortable list (shared group). onEnd → call move_task; on error revert DOM + frappe.show_alert.
  • Card: title, priority chip, assignee, deadline, risk flag. Inline-edit popovers for priority (select), assignee (team select), deadline (date) → patch_task. Click card body → open VT Task form.
  • Quick-add: + button per column header → inline title input → create_task → re-render column.

8. Testing

  • move_task: legal transition succeeds; illegal transition throws; Blocked toggle on/off restores PDCA status.
  • create_task: creates task in correct pdca_phase, defaults applied; access denied for non-member.
  • patch_task: allowed field updates; disallowed field rejected; access enforced.
  • project_board: grouping correct, allowed_targets matches transition table, access guard.

9. Out of scope (YAGNI)

  • Intra-column drag reorder / kanban_rank persistence — order is computed (priority→deadline).
  • Native Frappe Kanban Board doctype — rejected (route-coupled, no inline edit).
  • Re-opening a DONE task from the board (terminal phase; needs explicit reset op).
  • Legacy Cancelled tasks — skipped from the board, not shown.
  • Bulk actions, swimlanes, WIP limits.
