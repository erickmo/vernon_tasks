Onboarding Foundation — Design Spec

Date: 2026-06-01 · Status: Approved (design) · Task size: L · Branch: feat/onboarding-foundation

Make Vernon Tasks usable from zero. A brand-new user must be able to self-start: get a role, see a navigable product, understand the first action, and reach a working board on the first click. This spec covers the onboarding foundation scope — the five highest-impact root causes plus an optional demo seed and a hybrid native onboarding checklist. Out-of-scope items are listed explicitly.

Source of the problem set: full read-only UX/onboarding audit (7 agents, all 15 desk pages + flow), 2026-06-01.

Goals

  • A new authenticated user holds a usable role and never hits a bare permission wall on the landing page.
  • On a fresh deploy, the whole product is discoverable (navbar seeded), not two items behind typed URLs.
  • The primary landing path (home → project board) works on the first click.
  • Empty accounts present an inviting, action-oriented first-run experience with a clear "create your first X" call-to-action.
  • A guided, self-completing onboarding checklist appears exactly where the user lands (vt-home), backed by native, low-code, fixture-defined steps.
  • Users can optionally load and later remove demo data to see a live product without committing real data.

Non-Goals (deferred backlog — tracked from the audit, not built here)

  • VT Settings navbar-editor data-loss bug (audit #10).
  • Full Indonesian translation of the leader/exec/personal clusters.
  • Metric tooltips / jargon legends / kanban coachmarks (audit #6).
  • Filter-gated analytics auto-selection and email→name resolution (audit #8).
  • Shared card/layout standardization across exec/leader pages.

Key Architectural Decision — derive per-user completion, do not trust the native flag

Frappe's native Onboarding Step.is_complete / is_skipped are single fields on the step document, shared across all users (set via frappe.desk.desktop.update_onboarding_step → frappe.db.set_value("Onboarding Step", name, ...)). Native Module Onboarding is designed for one-time, single-admin setup, not per-contributor onboarding. If we relied on it, the first user to finish "Buat proyek" would mark it done for everyone.

Decision: Native Module Onboarding + Onboarding Step records are the declarative step catalog (titles, descriptions, action type, route/target) — low-code, fixture-exportable, admin-editable. Per-user completion is derived from the user's actual data at read time (does a brand exist, does the user have a project, does that project have ≥2 members, does the user have a task). This is correct per-user, self-completing, and needs no extra write path.

The native onboarding block still renders on the Vernon Tasks Workspace for the admin/native experience; the vt-home card is the per-user surface.

Architecture — 8 components

1. Auto-grant VT Member on login

  • Layer: Frappe framework event — on_session_creation hook (Hooks-First Priority 3).
  • Files: hooks.py (register hook); vernon_tasks/setup/roles.py (new, single documented function).
  • Behavior: On each session creation, if frappe.session.user is not Guest/Administrator and holds none of {VT Manager, VT Leader, VT Member}, add VT Member. Idempotent (role check first), cheap, covers new users, pre-existing role-less users, and users whose roles were revoked.
  • Why a hook, not a controller: there is no VT-owned doctype lifecycle for "a user logged in"; granting an app role on session start is a framework-event concern.

2. Auto-seed navbar on install and migrate (seed-if-empty)

  • Layer: Frappe framework events — after_install + after_migrate (Priority 3).
  • Files: hooks.py (register both); setup_website.py (new ensure_navbar_seeded()).
  • Behavior: If the VT Navbar Item child table is empty, run the existing seed; if it already has rows, do nothing (preserve admin customization). The existing setup_navbar_items() deletes-then-inserts and must NOT run unconditionally on migrate — ensure_navbar_seeded() wraps it behind the emptiness guard.
  • Result: a fresh deploy exposes all 14 seeded routes immediately instead of the DEFAULT_NAVBAR 2-item fallback.

3. Fix the broken landing→detail route

  • Layer: Client Script.
  • Files: task/page/vt_home/vt_home.js lines 126 and 141.
  • Change: frappe.set_route("vt-project", p.id) → frappe.set_route("vt-project-detail", p.id), matching vt_projects.js:50. Home project cards open the styled board, not the raw VT Project form (DocType slug collision).

4. Shared first-run empty-state component

  • Layer: Client Script, shared asset (Priority 4).
  • Files: public/js/vt_empty.js (new); registered in hooks.py app_include_js.
  • API: frappe.vt_ui.empty_state({ icon, title, message, cta_label, on_cta, secondary_label, on_secondary }) → returns a jQuery node rendering a dashed-border card (reusing the .vb-empty visual language) with an optional primary and secondary action.
  • Applied here: vt-home (single welcome card when led + member + next_actions are all empty) and vt-projects (Buat Proyek pertama Anda). One implementation, no duplication; reusable for the deferred pages later.

5. Primary "Buat" action above the fold on vt-home

  • Layer: Client Script.
  • Files: vt_home.js.
  • Change: page.set_primary_action("Buat Proyek", ...) opening a lightweight quick-create dialog (Nama Proyek + Brand; advanced fields deferred / progressively disclosed). Consistent with the vt-projects header action. Gives the landing page an obvious first action.

6. Hybrid onboarding checklist card on vt-home

  • Layer: Priority 5 whitelisted API (≤10-line entry, delegates) + Client Script + CSS.
  • Files: task/api/onboarding.py (new); vt_home.js (render card); vt_home.css (card styles).
  • API: get_onboarding_state() → returns the step catalog read from the Vernon Tasks Onboarding Module Onboarding record, each annotated with a derived is_complete for the current user, plus an overall dismissed flag.
  • Per-user completion derivation:
  • buat_brand — a VT Brand exists.
  • buat_proyek — the user leads or is a member of ≥1 VT Project.
  • tambah_tim — at least one of the user's projects has ≥2 team members.
  • buat_task — the user has ≥1 VT Task.
  • Card behavior: renders the checklist with progress (n/4), each row routes to the step's action (Go to Page / Create Entry). Hidden when all steps derive complete OR the user dismissed it. Dismiss persists per-user via frappe.defaults.set_user_default("vt_onboarding_dismissed", 1) and is read back in get_onboarding_state().
  • No write path needed for completion — derived state means the card self-updates as the user creates data.

7. Optional demo data (load + clear)

  • Layer: Setup module + whitelisted API entries (≤10 lines each, delegate).
  • Files: vernon_tasks/setup/demo_data.py (new, documented setup module analogous to setup_website.py); thin @frappe.whitelist() wrappers in task/api/onboarding.py.
  • load_demo(): creates 1 VT Brand (if none), 1 VT Project, 1 VT Sprint, and a few VT Tasks, all owned by the current user. Every created document's {doctype, name} is recorded in VT Settings.demo_data_refs (new JSON Small Text field on the single).
  • clear_demo(): deletes exactly the documents listed in demo_data_refs, then clears the field. Precise and reversible — never guesses by name.
  • Surface: a Muat data contoh secondary action inside the vt-home empty state / onboarding card; a Hapus data contoh action appears while demo refs exist.
  • Why a setup module, not a controller: demo seeding spans four doctypes and is a one-shot setup utility, not a single doctype's lifecycle — the same justification as the existing setup_website.py.

8. Native onboarding fixtures + workspace block

  • Layer: Layer 1 native config, exported as fixtures.
  • Files: hooks.py fixtures (add Module Onboarding, Onboarding Step, Onboarding Step Map filtered to the Vernon Tasks records); task/workspace/vernon_tasks/vernon_tasks.json (prepend an onboarding content block referencing the Module Onboarding).
  • Records (all labels Bahasa Indonesia): Module Onboarding "Vernon Tasks Onboarding" with four Onboarding Steps (Buat brand, Buat proyek pertama, Tambah anggota tim, Buat task pertama), each with an action (Create Entry / Go to Page) and route/target. These are the declarative catalog the vt-home card reads.

Onboarding steps (catalog)

Step key Label (ID) Action Target Derived "done" when
buat_brand Buat brand Go to Page vt-brands a VT Brand exists
buat_proyek Buat proyek pertama Create Entry VT Project user leads/member of ≥1 project
tambah_tim Tambah anggota tim Go to Page vt-team a user project has ≥2 members

Data model changes

  • VT Settings (single): new field demo_data_refs (Small Text, JSON list of {doctype, name}). Holds the demo documents created for precise teardown.
  • Per-user default vt_onboarding_dismissed (via frappe.defaults) — no schema change.
  • New native config records (fixtures): one Module Onboarding, four Onboarding Step, four Onboarding Step Map rows.

API surface (task/api/onboarding.py)

All @frappe.whitelist(), each ≤10 lines, delegating to setup/ helpers or doing a single read:

  • get_onboarding_state() → { steps: [{key, label, action, target, is_complete}], progress, dismissed, has_demo }.
  • dismiss_onboarding() → sets the per-user default; returns { ok: 1 }.
  • load_demo() → delegates to setup.demo_data.load; returns created counts.
  • clear_demo() → delegates to setup.demo_data.clear; returns removed count.

Error handling

  • load_demo() / clear_demo() wrap mutations in a single transaction; on failure roll back and return a friendly message, never a partial seed.
  • get_onboarding_state() tolerates a missing Module Onboarding record (returns an empty catalog → card simply not shown) so a half-migrated site does not error the landing page.
  • The vt-home card and empty-state render defensively: any API failure logs and falls back to the existing (fixed-routing) dashboard, never a blank page.

Testing plan (L → unit + integration + acceptance)

  • Unit: role-grant idempotency (no duplicate roles, skips Administrator/Guest); ensure_navbar_seeded is a no-op when rows exist and seeds when empty; each derived-completion predicate (brand/project/team/task) flips correctly; demo_data_refs round-trips and clear deletes exactly the recorded set.
  • Integration: get_onboarding_state returns correct per-user progress for a fresh user vs. a user with data; load_demo then clear_demo leaves the database in its prior state (no orphans).
  • Acceptance: brand-new user → login → lands on vt-home with VT Member, full navbar, onboarding card at 0/4; clicking a project card opens vt-project-detail; "Muat data contoh" lights up the dashboard; "Hapus data contoh" restores the empty state.
  • Test files tagged with the PRD/ADR IDs they cover; fixtures under tests/testdata/onboarding/.

Rollout

  1. Ship behind no flag (foundation is strictly additive and safe-by-default).
  2. After merge, run bench --site <site> migrate (triggers after_migrate seed-if-empty) and export-fixtures to commit the native onboarding records.
  3. Verify the acceptance path on a fresh test user.
