Portal Notifications P4c — Design Spec

Date: 2026-05-18 Parent PRD: docs/superpowers/specs/2026-05-17-desktop-portal-foundation-design.md Status: Draft — pending review Owner: Erick Mo

1. Scope

Desktop portal notification center — a bell-icon inbox in the portal TopBar that delivers real-time task and sprint activity to Manager, Leader, and Member users. P4c is the portal counterpart to P4a/P4b (mobile push notifications); it is NOT Web Push. All delivery is in-app, polling-based with optional socketio opt-in.

In scope (P4c): - Four notification sources: task assigned, task review result (approve/reject), sprint started/ended, mention/comment on user's task. - portal_notifications_enabled feature flag in VT Settings. - Backend: vernon_tasks/api/portal_notifications.py — new module, sibling to notifications.py. Reuses Vernon Notification DocType (already exists). - Backend: doc-event hooks in VT Task, VT Sprint, Comment for the four sources. - Frontend: NotificationBell component wired into TopBar.tsx. - Frontend: dropdown notification panel (latest 5 + "View all" link). - Frontend: dedicated full-page /portal/notifications for the inbox. - Permission model: Manager sees all scope notifications; Leader sees notifications scoped to their led projects/teams; Member sees own notifications only. - 30-second polling (consistent with PWA count_unread pattern); socketio opt-in noted in §2.1 as future path.

Non-goals (P4c, deferred): - In-portal notification preferences / mute rules (P4d). - Email digest triggered from portal notification queue. - Cross-user @mention parsing in comments (P4d — requires mention detection). - Web Push from portal (already handled by P4a/P4b; portal UI reuses same Vernon Notification rows without re-sending push). - Notification archival / bulk delete. - Realtime socketio delivery (opt-in path documented; not implemented here).

2. Architecture

2.1 Backend — doc events + notification queue

Doc events are registered in hooks.py under doc_events. Four hooks:

doc_events = {
    "VT Task": {
        "on_update": "vernon_tasks.api.portal_notifications.on_vt_task_update",
    },
    "Comment": {
        "after_insert": "vernon_tasks.api.portal_notifications.on_comment_insert",
    },
    "VT Sprint": {
        "on_update": "vernon_tasks.api.portal_notifications.on_vt_sprint_update",
    },
}

after_workflow_action is NOT used for approve/reject. Workflow action results surface as field changes on VT Task (review_status or equivalent field) in on_update — handled by the same on_vt_task_update handler via a field-delta check (see §3.2).

Socketio opt-in path (not implemented in P4c): After creating a Vernon Notification row, call frappe.publish_realtime(event="vt_notification", message={"user": user}, user=user). The frontend NotificationBell can then listen via frappe.realtime.on("vt_notification", ...) to trigger an immediate refetch. This path is a one-line addition when ready; polling covers P4c entirely.

Queue helper — queue_notification:

Defined in portal_notifications.py. All four source handlers call this single helper. Signature:

def queue_notification(
    user: str,
    event_type: str,
    reference_doctype: str,
    reference_name: str,
    message: str,
) -> None
  • Guards: skip if user == "Guest" or user == frappe.session.user (no self-notifications).
  • Deduplication: check frappe.db.exists("Vernon Notification", {"user": user, "event_type": event_type, "reference_doctype": reference_doctype, "reference_name": reference_name, "is_read": 0}) — if unread row already exists for same (user, event_type, reference_name), skip insert (prevents duplicate flood on rapid saves).
  • Insert with ignore_permissions=True; sets is_read=0.
  • After insert: call _invalidate_unread_cache(user) (cache key: vt:portal:notif:unread:{user}, TTL 30s, matches polling interval).

2.2 Backend — portal_notifications.py API

Module path: vernon_tasks/api/portal_notifications.py

Sibling to vernon_tasks/task/api/notifications.py (PWA). Whitelisted RPC endpoints exposed as vernon_tasks.api.portal_notifications.*:

list_notifications(limit, offset, only_unread, event_type_filter)

GET vernon_tasks.api.portal_notifications.list_notifications
    ?limit=20&offset=0&only_unread=0&event_type_filter=
→ {
    "results": [
      {
        "name": "VN-0001",
        "event_type": "task_assigned",
        "reference_doctype": "VT Task",
        "reference_name": "VT-0042",
        "message": "New task assigned: Fix login redirect",
        "is_read": 0,
        "creation": "2026-05-18 10:23:00"
      },
      ...
    ],
    "total_unread": 3
  }
  • limit: clamped 1–100 (reuse clamp_int from security.py).
  • offset: clamped 0–10,000.
  • only_unread: int 0/1 — adds is_read=0 filter when 1.
  • event_type_filter: optional string, one of {task_assigned, task_review, sprint_status, comment} — if provided, adds event_type= filter.
  • Permission scope filter (applied server-side before query):
  • Manager: no additional filter (all rows where user = session.user).
  • Leader: rows where user = session.user (notifications are already user-targeted; no cross-user read in P4c).
  • Member: same — user = session.user.
  • Note: Vernon Notification.user is always the recipient. Permission scope affects what notifications get created (§5), not what gets listed — each user only ever reads their own rows.
  • ORDER BY creation DESC, LIMIT limit OFFSET offset.
  • Returns total_unread from cache (same key as count_unread); avoids second COUNT query when cache is warm.

count_unread()

GET vernon_tasks.api.portal_notifications.count_unread
→ {"count": 3}
  • Cache key: vt:portal:notif:unread:{user}, TTL 30s.
  • Returns {"count": 0} for Guest without throwing (consistent with PWA).

mark_read(name)

POST vernon_tasks.api.portal_notifications.mark_read
Body: {"name": "VN-0001"}
→ {"ok": True}
  • Fetch doc; verify doc.user == frappe.session.user, else frappe.throw("Forbidden", frappe.PermissionError).
  • Set is_read=1, save(ignore_permissions=True).
  • Invalidate unread cache for session user.

mark_all_read()

POST vernon_tasks.api.portal_notifications.mark_all_read
Body: {}
→ {"ok": True}
  • frappe.db.set_value("Vernon Notification", {"user": user, "is_read": 0}, "is_read", 1, update_modified=False).
  • Invalidate unread cache.

get_feature_flag()

GET vernon_tasks.api.portal_notifications.get_feature_flag
→ {"enabled": true}
  • Reads VT Settings.portal_notifications_enabled. Cached for 60s (vt:portal:notif:flag).
  • Used by NotificationsFeatureGate on the frontend to conditionally render the bell.

TopBar.tsx change — replace the static bell button with NotificationsFeatureGate wrapping NotificationBell:

// Before:

// After:

NotificationBell.tsx: - Uses useNotificationCount — a React Query query with refetchInterval: 30_000 and staleTime: 25_000. - Renders a <button> with aria-label="Notifications" and aria-haspopup="dialog". - Shows an unread badge (<span className="notif-bell__badge">) when count > 0. Badge capped at 99+. - On click: toggle isPanelOpen state. Click-outside closes panel (use useClickOutside hook or equivalent). - Panel is rendered inline as a sibling via a <div> portal to document.body to avoid overflow clipping from topbar CSS. Uses position: fixed coordinates derived from bell button getBoundingClientRect().

NotificationPanel.tsx: - Fetches list_notifications(limit=5, offset=0) when panel opens (enabled: isPanelOpen). - Shows a loading skeleton (3 placeholder rows) while fetching. - Shows up to 5 NotificationItem rows. - Footer row: "Mark all read" button (calls mark_all_read, invalidates React Query cache) + "View all" <Link to="/portal/notifications">. - Empty state: "No notifications" illustration + text when results.length === 0 and count === 0. - Panel closes on Escape key and on navigation to /portal/notifications.

NotificationItem.tsx:

Props: notification: PortalNotification, onRead: (name: string) => void

  • Renders: icon (event-type-based), message string, relative time (creation formatted via date-fns/formatDistanceToNow).
  • Unread rows have data-unread="true" + CSS highlight.
  • Click: calls onRead(name) (optimistically sets is_read=1 in local cache) then navigates to the reference entity:
  • task_assigned / task_review / comment → /portal/projects?task=:reference_name (or task detail route if available).
  • sprint_status → /portal/projects/:projectId/sprints (derive projectId from reference_name via a VT Sprint lookup — pre-populated in the notification message or add a reference_parent field if needed — see §9).

hooks/useNotificationCount.ts:

export function useNotificationCount() {
  return useQuery({
    queryKey: ["portal", "notif", "count"],
    queryFn: () => portalNotificationsApi.countUnread(),
    refetchInterval: 30_000,
    staleTime: 25_000,
    select: (data) => data.count,
  });
}

hooks/useNotifications.ts:

export function useNotifications(params: {
  limit?: number;
  offset?: number;
  onlyUnread?: boolean;
  eventTypeFilter?: string;
  enabled?: boolean;
}) {
  return useQuery({
    queryKey: ["portal", "notif", "list", params],
    queryFn: () => portalNotificationsApi.listNotifications(params),
    enabled: params.enabled ?? true,
    staleTime: 15_000,
  });
}

3. Notification Sources

3.1 Task Assigned

Trigger: VT Task.on_update → on_vt_task_update handler.

Detection: Check doc.get_doc_before_save(). Fire only when doc.assigned_to != doc_before.assigned_to AND doc.assigned_to is not empty. This prevents spurious notifications on every save.

Recipient: doc.assigned_to

Self-notification guard: Skip if doc.assigned_to == frappe.session.user (person assigned themselves, or system process — still skip).

Payload to queue_notification:

queue_notification(
    user=doc.assigned_to,
    event_type="task_assigned",
    reference_doctype="VT Task",
    reference_name=doc.name,
    message=f"Task assigned to you: {doc.title}",
)

Message template: "Task assigned to you: {doc.title}"

Icon (frontend): clipboard / task icon.

3.2 Task Review (Approve / Reject)

Trigger: VT Task.on_update → on_vt_task_update handler.

Detection: Check field review_status (or the field that Frappe workflow sets on approve/reject — confirm field name from VT Task schema; if workflow uses workflow_state, watch that field instead). Fire when review_status changes to one of {Approved, Rejected} AND doc.assigned_to is not empty.

Recipient: doc.assigned_to (task owner receives the review result).

Self-notification guard: Skip if doc.assigned_to == frappe.session.user.

Payload:

action = "approved" if doc.review_status == "Approved" else "rejected"
queue_notification(
    user=doc.assigned_to,
    event_type="task_review",
    reference_doctype="VT Task",
    reference_name=doc.name,
    message=f"Your task was {action}: {doc.title}",
)

Message templates: - Approved: "Your task was approved: {doc.title}" - Rejected: "Your task was rejected: {doc.title}"

Icon (frontend): checkmark (approved) or X (rejected) — derive from message prefix or add a meta field (see §9).

3.3 Sprint Started / Ended

Trigger: VT Sprint.on_update → on_vt_sprint_update handler.

Detection: Check doc.get_doc_before_save(). Fire when doc.status changed to Active (sprint started) or Completed (sprint ended).

Recipients: All users who have tasks in this sprint — frappe.get_all("VT Task", filters={"sprint": doc.name}, pluck="assigned_to"). Deduplicate list. Filter out empty/Guest values.

Payload (started):

for user in recipients:
    queue_notification(
        user=user,
        event_type="sprint_status",
        reference_doctype="VT Sprint",
        reference_name=doc.name,
        message=f"Sprint started: {doc.sprint_title}",
    )

Payload (ended):

for user in recipients:
    queue_notification(
        user=user,
        event_type="sprint_status",
        reference_doctype="VT Sprint",
        reference_name=doc.name,
        message=f"Sprint completed: {doc.sprint_title}",
    )

Message templates: - Started: "Sprint started: {doc.sprint_title}" - Completed: "Sprint completed: {doc.sprint_title}"

Performance note: For large sprints (>200 tasks), the recipient fetch could be expensive. In P4c: accept O(n_tasks) lookup; log a warning if recipient count > 100. P4d can optimize with background job / celery queue.

Icon (frontend): lightning bolt (started) or flag (completed).

3.4 Mention / Comment

Trigger: Comment.after_insert → on_comment_insert handler.

Scope: Only for comments on VT Task documents. Check doc.reference_doctype == "VT Task".

Recipient: The assigned_to user of the referenced task — frappe.db.get_value("VT Task", doc.reference_name, "assigned_to"). This covers the "comment on your task" case.

Self-notification guard: Skip if doc.comment_by == recipient.

Payload:

commenter_name = frappe.db.get_value("User", doc.comment_by, "full_name") or doc.comment_by
task_title = frappe.db.get_value("VT Task", doc.reference_name, "title") or doc.reference_name
queue_notification(
    user=recipient,
    event_type="comment",
    reference_doctype="VT Task",
    reference_name=doc.reference_name,
    message=f"{commenter_name} commented on your task: {task_title}",
)

Message template: "{commenter_full_name} commented on your task: {task_title}"

Icon (frontend): speech bubble / comment icon.

Future @mention support (P4d): Parse doc.content for @username patterns → notify those users as additional recipients.

4. Notification Center UI

4.1 Bell icon in TopBar

  • Renders in the existing portal-topbar right zone, between search button and profile button (current DOM order unchanged).
  • Icon: uses an SVG bell icon, not an emoji. Replace the 🔔 placeholder.
  • Unread badge: absolute-positioned pill on the icon. Red background, white text. Hides when count === 0. Shows 99+ when count exceeds 99.
  • Badge animation: scale-in CSS keyframe on count change (subtle, not distracting).
  • Keyboard: focusable, Enter/Space opens panel.
  • aria-label updates dynamically: "Notifications — {count} unread" when unread > 0, "Notifications" when zero.

4.2 Dropdown panel

Dimensions: width: 380px, max-height: 480px, overflow-y auto on item list. Positioned below bell button, right-aligned to topbar edge.

Anatomy (top to bottom): 1. Header row — "Notifications" heading (left) + "Mark all read" text button (right). Mark all read only renders when total_unread > 0. 2. Item list — up to 5 NotificationItem rows. Each row: - 40×40 icon area (event-type icon, colored circle background). - Message text (1–2 lines, overflow ellipsis on line 2). - Relative timestamp (e.g., "3 minutes ago") — bottom-right. - Unread indicator: left border accent (3px, brand color) + slightly lighter row background. - Click navigates to reference entity AND marks read (optimistic). 3. Footer row — "View all notifications →" link to /portal/notifications. Always shown. 4. Empty state (shown when results.length === 0) — centered icon + "You're all caught up" text.

Panel shadow: box-shadow: 0 8px 24px rgba(0,0,0,0.12). Border-radius: 8px. Border: 1px solid var(--portal-border).

Loading state: 3 skeleton rows (animated shimmer), same height as real rows.

Error state: "Could not load notifications. Retry?" with a retry button that calls refetch().

4.3 Full notifications page (/portal/notifications)

URL: /portal/notifications

Nav entry: Add "Notifications" link to portalNav in nav.ts with key notifications, path /portal/notifications, visible to all roles. The nav item shows an unread count badge inline (uses same useNotificationCount hook).

Page layout: - Standard PageLayout wrapper (same as OKR/Projects pages). - Page title: "Notifications" - Subheading: active filter description (e.g., "Showing unread tasks reviews").

Filter bar:

[All] [Tasks] [Reviews] [Sprints] [Comments]    [□ Unread only]    [Mark all read]
  • Tabs map to event_type_filter values: "", "task_assigned", "task_review", "sprint_status", "comment".
  • "Unread only" toggle maps to only_unread=1.
  • Active tab highlighted with border-bottom accent.

Item list: - Full-width NotificationItem rows with no message truncation. - Grouped by date (Today / Yesterday / Older) using sticky date dividers. - Read rows dimmed (opacity 0.6) but still clickable. - "Load more" button below list when server returns exactly limit rows (implies more pages). Appends next page to list.

Empty state per filter: - All + no notifications: "Nothing here yet. Notifications will appear when tasks are assigned, reviewed, or sprints change." - Filtered: "No {filter_label} notifications."

5. Permissions

Permission logic is enforced server-side in the notification creation hooks (what gets enqueued) and in the list endpoint (what gets returned). The frontend enforces nothing beyond hiding the feature when the flag is off.

5.1 Notification creation scope

Event Manager receives Leader receives Member receives
task_assigned Yes, when assigned_to = self Yes, when assigned_to = self Yes, when assigned_to = self
task_review Yes, when assigned_to = self Yes, when assigned_to = self Yes, when assigned_to = self
sprint_status Yes, if has task in sprint Yes, if has task in sprint Yes, if has task in sprint
comment Yes, when task assigned_to = self Yes, when task assigned_to = self Yes, when task assigned_to = self

All notification rows are user-targeted (the user field is the recipient). No row is ever visible to a user other than its intended recipient via the portal API.

5.2 List endpoint scoping

list_notifications filters by user = frappe.session.user unconditionally. There is no Manager-reads-all-users mode in P4c — each user sees only their own notifications. A future Manager notification feed (P4d) would be a separate endpoint.

5.3 Feature flag gate

  • Backend: every whitelisted endpoint in portal_notifications.py checks frappe.db.get_single_value("VT Settings", "portal_notifications_enabled") at entry. If 0, raise frappe.PermissionError("Portal notifications are not enabled"). Cache the flag value for 60s.
  • Frontend: NotificationsFeatureGate calls get_feature_flag() on mount. If enabled=false, renders null (bell is invisible, route returns <NotFound />).

6. Testing

6.1 Backend — tests/portal/test_portal_notifications.py

queue_notification unit tests: - Happy path: creates Vernon Notification row with correct fields. - Self-notification guard: frappe.session.user == recipient → no row created. - Guest guard: user == "Guest" → no row created. - Deduplication: second call with identical (user, event_type, reference_name) + is_read=0 → no second row. - After insert: count_unread cache is invalidated.

on_vt_task_update tests: - assigned_to changes from empty to user A → notification created for user A. - assigned_to changes from user A to user B → notification created for user B, NOT user A. - assigned_to unchanged → no notification. - review_status changes to Approved → notification to assigned_to. - review_status changes to Rejected → notification to assigned_to. - review_status unchanged on save → no notification.

on_vt_sprint_update tests: - Status changes to Active → notification sent to all task owners in sprint. - Status changes to Completed → notification sent to all task owners. - Status changes to Planning (non-trigger) → no notification. - Sprint with no tasks → no notification (empty recipient list).

on_comment_insert tests: - Comment on VT Task → notification to task assigned_to. - Comment by same user as assigned_to → no notification (self-comment guard). - Comment on non-VT Task doctype → no notification.

list_notifications endpoint tests: - Returns only rows for session.user. - only_unread=1 filter: returns only is_read=0 rows. - event_type_filter="task_assigned" narrows results. - Pagination: offset=20, limit=5 returns correct slice. - total_unread matches live count.

mark_read tests: - Marks correct row; another user's row raises PermissionError. - Cache invalidated after mark.

mark_all_read tests: - All unread rows for user set to is_read=1. - Another user's rows unaffected. - Cache invalidated.

Feature flag tests: - All endpoints raise PermissionError when portal_notifications_enabled=0.

  2. Backend deploy — portal_notifications.py module + doc event hooks in hooks.py. Smoke test: confirm all endpoints return PermissionError when flag is 0. Confirm no existing functionality broken.
  3. Frontend deploy — NotificationsFeatureGate renders null when flag is 0. Bell is invisible. No user-visible change.
  4. Pilot org — flip portal_notifications_enabled=1 on a single site. Smoke test the four notification flows:
  5. Assign a task → bell count increments within 30s → open panel → notification visible → click navigates correctly.
  6. Leader approves/rejects a task → member sees review notification.
  7. Start a sprint → all task members see sprint notification.
  8. Add a comment to a task → assigned member sees comment notification.

9. Open questions / risks

  1. review_status field name — Spec assumes VT Task has a review_status field (or workflow_state) that changes to Approved/Rejected. Confirm exact field name before implementing on_vt_task_update. If the field is named differently, the detection condition in §3.2 must be updated.

  2. Sprint navigation from notification — NotificationItem click for sprint_status events needs to navigate to /portal/projects/:projectId/sprints. The reference_name is a VT Sprint name, not the project ID. Two options:

  3. Option A: During queue_notification, append project_id into the message field or add a reference_parent field to Vernon Notification. Cleaner but requires schema change.
  4. Option B: Frontend fetches the project for the sprint lazily on click. Extra API call but no schema change.

  5. Recommendation: Option A — add reference_parent (Data, optional) to Vernon Notification in a patch. Low cost, cleaner navigation.

  6. Sprint fanout at scale — on_vt_sprint_update queries all task owners in a sprint. For a 50-person sprint, this inserts 50 rows synchronously in the doc event handler. Acceptable for P4c; defer to async queue in P4d if sprint size > 100 becomes common.

  7. Vernon Notification DocType pre-existing content — P4a/P4b may have written rows using different event_type values (or may use Notification Log entirely). Verify which DocType and which field naming convention P4a/P4b actually used, to ensure P4c rows use consistent values. If P4a/P4b used Notification Log instead of Vernon Notification, P4c should align or the frontend must unify both sources.

  8. doc.get_doc_before_save() availability — This Frappe method returns the pre-save state from frappe.db.get_doc cache. Confirm it is reliably populated in on_update for VT Task and VT Sprint. If unreliable, alternative: check doc.has_value_changed("assigned_to") (Frappe v15 API) which is more reliable.

  9. Concurrent mark-all-read race — If the user opens the panel, reads 3 items via individual clicks, then clicks "Mark all read" concurrently while a background poll is in flight, the unread count could briefly show stale data. Mitigation: markAllRead mutation's onSuccess callback calls queryClient.invalidateQueries(["portal", "notif"]) to refetch both count and list. Acceptable flicker (< 1 poll cycle = 30s).

  10. Socketio opt-in activation — If realtime delivery is desired before P4d, the one-line addition in queue_notification (§2.1) plus a frappe.realtime.on handler in useNotificationCount is sufficient. No architecture change needed; document the path clearly for the P4d implementer.
