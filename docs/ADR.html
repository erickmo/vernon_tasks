ADR-001 — Build on Frappe Framework

Status: Accepted Date: 2026-04 (project inception) Deciders: Vernon Tasks core team
Context

Need company-wide task / OKR / sprint tool integrated with existing ERPNext stack. Users, roles, permissions, audit, email already in Frappe. Building greenfield = duplicate identity + role mgmt + reporting.

Decision

Build Vernon Tasks as Frappe v15 app. Reuse Frappe User, Notification Log, permissions, scheduler, Report Builder, REST. Add domain doctypes only.

Consequences
  • + Zero auth/role work; native CRUD via /api/resource/.
  • + Built-in workspaces, reports, fixtures, scheduler.
  • + Hooks (doc_events, scheduler) cover most cross-cutting needs.
  • − Tied to Frappe upgrade cadence and Python 3.11+.
  • − Frappe's Jinja/JS Desk has dated UX → motivates PWA (see ADR-002).
Alternatives
  • Django + DRF: full control but rebuild auth + roles + reporting.
  • SaaS (Asana, Linear): no OKR/KPI integration; vendor lock-in; no PDCA model.

ADR-002 — Dual UI: Frappe Desk + Mobile PWA

Status: Accepted Date: 2026-05-11
Context

ICs work mostly on phones; leaders + execs on desktop with rich analytics. Single UI fights both. Frappe Desk = good for power users; bad for mobile-first daily check-in.

Decision

Keep 7 Frappe Desk pages for desktop power users + leaders / execs. Ship installable React PWA at /m/ for mobile-first IC and on-the-go leader review.

Consequences
  • + Best UX per surface; mobile = native-feel, Desk = full bulk power.
  • + Backend stays single source of truth; both UIs hit same whitelisted APIs.
  • − Double surface to test (Vitest/Playwright + bench tests).
  • − Risk of UI drift; mitigated by ADR-016 (responsive nav).
Alternatives
  • Mobile-friendly Desk only: poor mobile UX, no offline, no install.
  • Native iOS/Android: cost, distribution, build pipeline.

ADR-003 — React 18 + Vite 5 + TypeScript 5 for PWA

Status: Accepted Date: 2026-05-11
Context

Need fast HMR, small prod bundle, mature ecosystem (router, query lib, charts), type safety, easy testing.

Decision

React 18 + Vite 5 + TS 5. react-router-dom@6, @tanstack/react-query@5, idb-keyval, Workbox 7, recharts (lazy).

Consequences
  • + ~300 ms dev reload; tree-shaken prod; ~95 KB gz main bundle.
  • + Concurrent features (Suspense for lazy chunks).
  • − Adds Node tooling step (Vite build) to deploy pipeline.
  • − Recharts cost mitigated by lazy split (see ADR-011).
Alternatives
  • Vue/Svelte: smaller bundle but smaller ecosystem for Frappe consultants.
  • Next.js: server-rendering not needed; SPA served by Frappe.
  • Preact: bundle win but loses React-ecosystem libs.

ADR-004 — Workbox StaleWhileRevalidate offline strategy

Status: Accepted Date: 2026-05-11
Context

Mobile field users hit spotty networks. Need fast first paint + fresh data when online + readable when offline. Full offline-first w/ sync queue = high complexity, hard for write conflicts.

Decision

Workbox 7 generateSW. Precache app shell. Runtime StaleWhileRevalidate on /api/method/vernon_tasks.* with 1-day cache expiry. Mutations rejected when offline; UI disables buttons + toast.

Consequences
  • + Cached lists render instantly; fresh fetched in background.
  • + Offline reads always work; mutations fail gracefully.
  • + No conflict-resolution complexity.
  • − No queued offline writes (deferred to future ADR if needed).
  • − 1-day staleness window for cached responses (mitigated by SWR refresh).

ADR-005 — Reuse Frappe session cookie (no JWT)

Status: Accepted Date: 2026-05-11
Context

PWA hosted same-origin as Frappe (/m/). Building separate JWT system = duplicate user store, refresh-token rotation, key mgmt. Frappe already issues httpOnly sid cookie.

Decision

PWA logs in via POST /api/method/login. Receives sid cookie. All subsequent API calls send cookie automatically. CSRF token fetched via /m/_boot on app start; required on mutations.

Consequences
  • + Zero auth surface to build; native Frappe role checks work as-is.
  • + Single sign-out across Desk + PWA.
  • − Same-origin only; no third-party PWA host.
  • − Session expiry → relogin modal (preserves route state).

ADR-006 — TanStack Query 5 for client cache

Status: Accepted
Context

Need declarative server-state cache with stale-while-revalidate, optimistic updates, invalidation on mutate.

Decision

Use @tanstack/react-query@5 as sole client-cache layer. Query keys: ["my-work"], ["task", id], ["search", q], etc. Stale time 5 min. Invalidate on mutation success.

Consequences
  • + No Redux/Zustand boilerplate; UI state via React local state.
  • + Optimistic updates trivial via onMutate + onError rollback.
  • − Two cache layers (RQ in-memory + Workbox SW); coordination via SWR semantics.

ADR-007 — Strict layer separation: Controller / Service / Page

Status: Accepted
Context

Frappe doctype controllers and Page handlers tend to absorb business logic → untestable, duplicated, fragile under refactor. Vernon adds ~20 doctypes; risk of God classes.

Decision
  • Controller (doctype/<name>.py): validation, db_set, lifecycle hooks. No math.
  • Service (task/services/*.py): logic, calculations, algorithms. Pure when possible. DI for settings.
  • Page handler (page/<name>/<name>.py): auth check, request parse, call services, format response.

Function length ≤ 40 lines. No class > 300 lines / 5 responsibilities. Named constants only.

Consequences
  • + Services unit-testable without Frappe boot.
  • + Easy onboarding for new devs.
  • − Slightly more files; navigation discipline required.

ADR-008 — PDCA as first-class state machine on Task & Project

Status: Accepted
Context

Free-form status fields drift; can't enforce review gates. Vernon's gamified points only fair if every Task funnels through review.

Decision

Define pdca_phase field with strict transition matrix per doctype. Validator rejects illegal transitions. Kanban Status derived/mapped 1:1 from PDCA Phase.

VT Task: BACKLOG → PLAN → DO → CHECK → DONE
                          ↑     │
                          └─ ACT┘

Internal services use db_set("pdca_phase", phase, update_modified=False) to bypass validation only in trusted code paths.

Consequences
  • + Audit-friendly: every state change traceable.
  • + Leader review = enforced bottleneck for points.
  • − More transitions to enumerate; some flows verbose.

ADR-009 — Immutable Task Point Log for all point math

Status: Accepted
Context

Points feed compensation discussions / leaderboards. Computing from current task fields = no audit; manager overrides leave no trail.

Decision

Every point delta writes append-only row to Task Point Log: task, type (earned/early_bonus/late_penalty/revision_deduction/leader_override), amount, by_user, reason, timestamp. Monthly aggregate cached to User Point Summary.

Consequences
  • + Full audit. Overrides visible to all stakeholders.
  • + Reconstructable totals if cache lost.
  • − Storage cost grows linearly; mitigated by indexed reads + monthly rollup.

ADR-010 — Web Push (VAPID) over native shells (Capacitor)

Status: Accepted
Context

Need push for review queue / assignment / due reminders. iOS Web Push works only on PWA installed via A2HS (iOS 16.4+). Native shell = build pipeline, app store distribution.

Decision

Use Web Push API with VAPID keys generated once via bench vernon-generate-vapid → stored in VT Settings. Backend doctype Vernon Push Subscription. Notification Log.after_insert hook dispatches push. Dead endpoints (404/410) auto-pruned.

Consequences
  • + Zero app-store overhead.
  • + One install path: A2HS.
  • − iOS users must install PWA (≥16.4); modal hints user.
  • − No background sync APIs on iOS (acceptable for current scope).

Re-evaluate if iOS adoption is blocked → ADR-future for Capacitor.

ADR-011 — Recharts for charts, lazy-loaded

Status: Accepted
Context

Burndown, velocity, KPI trend, leaderboard all need responsive interactive charts. Recharts is React-idiomatic but heavy (~357 KB raw).

Decision

Use Recharts. Analytics + Leader pages lazy-loaded via React.lazy. Recharts shared chunk loaded only when entering chart pages. Main bundle pays nothing.

Consequences
  • + Main bundle stays ~95 KB gz.
  • + Chart-heavy pages acceptable load on Wi-Fi.
  • − First chart-page visit = larger payload; cached after.
Alternatives
  • Chart.js: imperative, harder to compose with React.
  • visx/d3 from scratch: too much custom code.

ADR-012 — Allowlisted client telemetry, 90-day retention

Status: Accepted
Context

Need observability for rollout (error rate, install rate, funnel) without external SaaS (privacy + cost). Free-form events risk PII leakage.

Decision

Client calls task.api.telemetry.log_event(event, props). Server enforces allowlist of ~40 event names. Rate-limit 60/min/user. Storage in Vernon Telemetry Event doctype, daily purge ≥90 days. Report Builder for funnels.

Consequences
  • + All data stays in Frappe; no third-party.
  • + Allowlist prevents accidental PII fields.
  • − No real-time dashboards (acceptable; ops use Report Builder).

ADR-013 — Pre-rollout hardening via security.py guards

Status: Accepted
Context

Before company-wide rollout (CSO sign-off), PWA APIs unbounded → potential abuse (brute mutations, large payloads, clickjack on SPA shell).

Decision

Add task/api/security.py with four reusable guards: require_login, rate_limit, clamp_int, max_str. Apply per-endpoint table. Inject CSP / X-Frame / Referrer / Permissions headers on /m/* responses.

Consequences
  • + Defense in depth additive layer; no business logic change.
  • + Reusable in future endpoints.
  • − Slight per-request overhead (Redis incr). Acceptable.

Out of scope: WAF, DocType permission audit, frontend validation (defense in depth).

ADR-014 — Phased PWA rollout P0.5 → P4b behind boot flags

Status: Accepted
Context

Big-bang PWA = high risk; pilot data needed before company-wide. Need backout per feature.

Decision

Sequential additive phases: P0.5 (foundation) → P1a (mutations + install) → P1b (search + notif) → P2 (dashboard + IC analytics) → P3a (leader review) → P3b (leader sprint + exec) → P4a (push) → P4b (push prefs). Each phase behind feature flag exposed via /m/_boot. Pilot gates: error_boundary < 1 %, login_failure < 5 %, install rate ≥ 30 %, no P0 bugs.

Consequences
  • + Clear ship/measure/decide loop.
  • + Per-feature backout via flag.
  • − Coordination overhead per phase; mitigated by spec+plan template.

ADR-015 — Optimistic UI w/ 5s undo; no offline mutation queue

Status: Accepted
Context

"Complete task" must feel instant on mobile. Real offline write-queue needs conflict resolution + sync layer.

Decision

useUndoableMutation shows toast w/ Batalkan button; setTimeout 5 s, then sends real request. If offline → reject up-front, toast "Sambungkan internet dulu". No queued retries.

Consequences
  • + Sub-100 ms perceived latency on completes.
  • + Simple mental model; no conflict edge cases.
  • − No "do it offline, sync later" UX (acceptable for current users).

ADR-016 — One PWA codebase serves both mobile + desktop (breakpoint 768px)

Status: Accepted Date: 2026-05-14
Context

Leaders + execs working from laptops want same data as mobile PWA. Building separate desktop UI = duplicate effort.

Decision

Single React codebase. useMediaQuery(768) hook drives layout. <768 px = BottomNav (5 tabs). ≥768 px = TopNav two-tier (Nav1 + Nav2 submenu). Page layouts adapt: Dashboard 2-col grid, Work master-detail ≥900 px. Tab state via useSearchParams for deep-link.

Consequences
  • + Single shipped artifact; one test matrix (+ viewport).
  • + Desktop users get install + offline + push for free.
  • − Layout complexity per page; mitigated by composable layout primitives.

ADR-017 — id-ID formal locale ("Anda") as baseline

Status: Accepted
Context

Vernon Corp is Indonesia-based; primary user language = Bahasa Indonesia. Mixed en/id UI creates cognitive load.

Decision

All user-facing strings = id-ID formal ("Anda", not "kamu"). Dates DD MMM YYYY (e.g., 16 Mei 2026), times HH:mm. Strings centralized in i18n.ts. English left only in code identifiers + dev docs.

Consequences
  • + Consistent professional tone.
  • + Lower friction for non-English users.
  • − Multi-locale support deferred (no urgent need).

ADR-018 — Git-SHA service-worker cache key

Status: Accepted
Context

Service workers infamously cache stale assets after deploy. Manual cache versioning forgotten → users stuck on old code.

Decision

Vite build injects VITE_GIT_SHA into SW cache name. Workbox generateSW uses it as cacheId. Every deploy = new SHA = new cache namespace; old cache evicted on activate.

Consequences
  • + Cache-busting automatic.
  • + Rollback possible: revert SHA → SW serves prior cache.
  • − Slight cold-load cost on every deploy (one fresh fetch).
