Brand Detail — lebih informatif

Date: 2026-06-06 Task size: M (extend 1 read endpoint, extract 1 reused helper, rewrite 1 page JS, add CSS, 2 test files, docs; ~6 files) App ownership: 100% vernon_tasks — no Frappe core files touched Base branch: master Builds on: docs/superpowers/specs/2026-06-04-brand-detail-okr-design.html

1. Goal

The vt-brand-detail page today shows only the OKR surface: a brand hero (logo / name / description) and collapsible period sections listing Objectives and their Key Results. It answers "what objectives, what KRs" but gives no at-a-glance health, hides data it already loads (confidence, objective_owner), and never links the brand's strategy to its execution (projects / sprints / tasks).

This change makes the page informative at a glance by adding four blocks inside the existing enriched-scroll layout:

  1. Summary stat bar — objective count, KR count, average progress, At Risk count, active-period progress, and a status-distribution segment bar.
  2. Execution section — the brand's projects + active sprint + remaining work + effort-weighted progress, reusing the exact rollup that powers the /app/vt-brands cards.
  3. Owner on objective cards — avatar + name (data already linked, just not rendered).
  4. Confidence on KR rows — confidence% chip (already in payload, just not rendered).

Native Frappe infrastructure only: one whitelisted read endpoint, extended; no new page, no new doctype, no core files touched.

2. Non-goals (YAGNI)

  • No multi-period sparkline chart. Trend is expressed as a progress % per period header + the active-period bar — honest with available data, no chart library.
  • No new write paths. All create/edit dialogs stay exactly as they are.
  • No second HTTP round-trip. Everything ships in the existing single get_brand_okr call.
  • No per-project drill-down page. The execution list links to existing vt-project-detail.

3. Architecture

3.1 Read API — extend brand/api/brand_okr.py

Keep the single endpoint get_brand_okr(brand_id). Add three things to its response, each delegated to a helper so the entrypoint stays under 40 lines.

3.1.1 summary — zero extra queries

Computed from the objectives + KRs already loaded in memory. No DB hit.

"summary": {
  "objective_count": 12,
  "kr_count": 34,
  "avg_progress": 68,
  "status_counts": {"Open": 3, "On Track": 6, "At Risk": 2, "Closed": 1},
  "at_risk_count": 2,
  "active_period": {"period": "2026-Q2", "progress": 72}
}
  • avg_progress = mean of per-objective aggregate progress (0 when no objectives).
  • status_counts drives the segment bar; at_risk_count is a convenience mirror of the "At Risk" bucket.
  • active_period = the period flagged is_current (null when none); progress = mean of that period's objective progress.

3.1.2 execution — reuse portal_brands rollup (no duplication)

The per-brand sprint/task/progress rollup already exists in brand/api/portal_brands.py (_task_aggregates, _active_sprints, _progress_pct) but is private and computes the map for all brands. Extract a single-brand public function there and call it from brand_okr.py:

# portal_brands.py
def brand_execution(brand_id: str) -> dict:
    """Single-brand sprint/task/progress rollup + project list.
    Reuses the same _task_aggregates / _active_sprints / _progress_pct
    primitives as the brand-list cards so the numbers cannot drift."""

Response block:

"execution": {
  "project_count": 4,
  "active_sprint_count": 1,
  "active_sprint_title": "W-23 Sprint",
  "remaining_tasks": 18,
  "remaining_minutes": 240,
  "total_minutes": 600,
  "progress_pct": 60,
  "projects": [{"id": "PROJ-0001", "name": "Situs Baru", "progress": 60}]
}

The same completion contract holds: kanban_status-based done detection, Cancelled work excluded from both numerator and denominator, effort-weighted progress with a task-count fallback when no estimates exist.

3.1.3 owner display + per-period progress

  • Batch-fetch User.full_name + user_image for every distinct objective_owner in one frappe.get_all (avoids N+1). Attach owner_name + owner_image onto each objective (fall back to the raw owner id / initial when missing).
  • Add progress to each period dict = mean of its objectives' aggregate progress, for the period header label.

3.2 UI — task/page/vt_brand_detail/vt_brand_detail.js

Enriched-scroll layout. Order: hero → stat bar → execution section → period sections. New render functions, each focused and small:

  • stat_bar(summary) — 4 metric chips [Obj][KR][avg%][At Risk] + an "Period aktif: NN%" progress line + a thin segmented status bar (On Track green / At Risk amber / Closed gray / Open slate). Renders nothing meaningful-but-empty when counts are zero.
  • execution_section(brand_id, execution) — collapsible (mirrors period_section caret pattern). Shows active sprint title + count, remaining tasks/minutes, a progress bar, and the project list (each row links to vt-project-detail). Hidden entirely when project_count === 0.
  • objective_card — add an owner chip (avatar img or initial circle + name) into the existing header row.
  • kr_row — add a confidence chip c:NN% next to the progress %.
  • period_section header — append the period progress % beside the objective count.

All new strings are Bahasa Indonesia. All values run through the existing esc() / pct() helpers. No business logic in the page — it only renders the payload.

3.3 CSS — public/js/vt_home.css (or the page's stylesheet)

Move repeated inline styles into named classes to keep the JS readable: .vt-stat-bar, .vt-stat-chip, .vt-stat-chip--risk, .vt-status-seg, .vt-owner-chip, .vt-kr-conf, .vt-exec. No magic colors inline — define them once.

4. Data flow

vt_brand_detail.js  on_page_load
  -> frappe.call get_brand_okr(brand_id)        [ONE request]
       brand_okr.get_brand_okr
         _read_objectives / _read_key_results   (existing)
         _summary(objectives, krs_by_obj)       (new, in-memory)
         portal_brands.brand_execution(brand_id)(new, reused primitives)
         _owner_map(objectives)                 (new, 1 query)
  -> render: hero, stat_bar, execution_section, period_sections

7. Risks & mitigations

  • Number drift between brand-list cards and detail execution → mitigated by reusing the same primitives + a no-drift test asserting equality with _brand_stats_map().
  • Restart needed: extending an existing whitelisted method does not need a new registration, but gunicorn caches the module — restart frappe-backend-1 after deploy (see memory: new whitelist method needs backend restart).
  • Owner image missing → initial-circle fallback, same as the brand-logo fallback.
