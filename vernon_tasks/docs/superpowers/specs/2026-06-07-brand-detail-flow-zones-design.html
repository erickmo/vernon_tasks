1. Goal

Make vt-brand-detail express the verified domain flow end to end:

VT Brand ‚îÄ‚î¨‚îÄ [STRATEGI]  Objective ‚îÄ‚îÄ< Key Result        (OKR tree, mandatory)
          ‚îÇ              KPI Definition ‚îÄ‚îÄ< KPI Entry     (KPI tree, mandatory)
          ‚îî‚îÄ [EKSEKUSI]  VT Project ‚îÄ‚îÄ< VT Sprint ‚îÄ‚îÄ< VT Task

The two trees both hang off Brand by mandatory links; they are stitched by optional bridges (VT Project.objective, KPI Entry.project). The page today shows the OKR surface + an execution rollup but (a) renders no KPI at all, and (b) never shows the OKR‚ÜîProject bridge. This change reorganizes the page into two labeled zones and surfaces both the KPI tree and the bridges.

Native Frappe infrastructure only: one whitelisted read endpoint extended, one doctype field added (app-owned doctype, not a Custom Field), no new page.

  • Read API: brand/api/brand_okr.py::get_brand_okr ‚Äî returns brand, perms, summary, execution (via brand_execution), periods[].objectives[].key_results[].
  • brand/api/brand_strategy.py::get_brand_strategy_tree ‚Äî already computes Brand ‚Üí Objective ‚Üí {KRs, KPI Definitions, linked projects} + unlinked projects, but is wired to nothing (orphaned; no JS/page/test references it).
  • KPI Definition fields: kpi_name, brand(reqd), frequency(reqd), allow_negative, unit, objective(opt), formula. No target field. KPI Entry: kpi_definition (reqd), date(reqd), value(reqd), project(opt), notes.

3. Data model change

Add one field to the app-owned KPI Definition doctype (okr/doctype/kpi_definition/kpi_definition.json):

Field Type Reqd Notes
target_value Float 0 (optional) Goal value for the metric. Optional so existing KPI Definitions keep working.
  • Optional, not mandatory ‚Äî existing rows have no target.
  • Migration: bench --site task.localhost migrate (Frappe auto-adds the column).
  • Not a fixture: KPI Definition is not in hooks.py fixtures, so no fixture export step.
  • Controller (kpi_definition.py::validate): when target_value is set and allow_negative is 0, reject a negative target (mirror the existing positivity intent). One inline guard, no new method.

KPI progress = clamp(latest_value / target_value * 100, 0, 100) when target_value > 0; when target is unset, progress is None and the row renders value-only (no bar). allow_negative KPIs with a target still clamp display to [0,100] for the bar; the raw value is always shown alongside.

4. API change ‚Äî brand_okr.py

get_brand_okr gains two payload blocks and one execution-field addition. All additions are batched to preserve the single-round-trip, no-N+1 contract.

4.1 Bridge ‚Äî projects per objective

New _read_objective_projects(objective_ids) ‚Üí one frappe.get_all("VT Project", filters={objective: ["in", ids]}, fields=[name, title, status, percent_done, objective]), grouped by objective. Each objective in periods[].objectives[] gains:

"projects": [{"id", "title", "progress" (round percent_done), "status"}]

4.2 KPI tree ‚Äî brand-level KPI block

New _read_brand_kpis(brand_id): 1. frappe.get_all("KPI Definition", filters={brand}, fields=[name, kpi_name, unit, frequency, objective, target_value, allow_negative]). 2. frappe.get_all("KPI Entry", filters={kpi_definition: ["in", def_ids]}, fields=[kpi_definition, date, value], order_by="date desc") ‚Äî take the latest two entries per definition (latest + previous) in Python. 3. Emit kpis: [{id, kpi_name, unit, frequency, objective, objective_title, target_value, latest_value, latest_date, prev_value, progress_percent, trend}] where trend ‚àà {up, down, flat, none} from latest vs prev, and progress_percent per ¬ß3.

Returned at top level: "kpis": [...]. Added to summary: kpi_count.

4.3 Execution bridge

portal_brands.py::brand_execution projects gain objective + objective_title (one extra get_all field + a batched Objective-title lookup), so EKSEKUSI rows can show which objective each project serves.

4.4 Permissions

Add can_read_kpi = has_permission("KPI Definition", "read"). KPI block hidden when false. No KPI mutation in this change (KPIs are managed on their native forms) ‚Äî read-only surface.

4.5 Cleanup

Delete brand/api/brand_strategy.py (orphaned; no test file exists). Its query patterns (objective‚Üíprojects, objective‚Üíkpi_defs) are reused inside brand_okr.py helpers above ‚Äî one source of truth.

9. Risks & mitigations

  • Backend restart: extending an existing whitelisted method needs no new registration, but gunicorn caches the module ‚Äî restart frappe-backend-1 after deploy (memory: new whitelist method needs backend restart). The new doctype field needs bench migrate + a restart.
  • N+1: all new reads are batched ["in", ids] queries ‚Äî assert query count stays flat in the API test.
  • Existing KPI rows without target: render value-only; no crash, no fake 0% bar ‚Äî progress_percent is None path tested.
  • Deleting brand_strategy.py: confirmed orphaned (no JS/page/test reference); grep before delete in the plan.
  • allow_negative KPIs: bar clamps to [0,100] but raw value always shown, so a negative-trending KPI is not misread as 0.
