Merge "My Work" page into vt-home

Date: 2026-06-03
Task size: L (deletes a desk Page + module, moves 5 whitelisted APIs, new migration patch, retargets tests; >10 files)
App ownership: 100% vernon_tasks — no Frappe core files touched
Scope: Fold the my-work desk Page (My Day / What To Do Today / My Blocked Tasks + Start / Submit-for-Review actions) into vt-home as a third tab "Tugas Saya", then retire the /app/my-work route entirely.

1. Goal

Collapse two overlapping personal dashboards into one. Today a user lands on vt-home (Beranda + Tim tabs) but the focused "doer" workflow (My Day, What To Do Today, My Blocked Tasks, with one-click Start / Submit-for-Review) lives on a separate my-work page reached via a nav link. After this change there is a single landing page with three tabs — Beranda | Tugas Saya | Tim — and the standalone my-work route is removed. Native Frappe infrastructure only; no core files touched.

2. Locked decisions

DecisionChoice

3. Frontend — "Tugas Saya" tab

vt-home renders three tabs: Beranda (always), Tugas Saya (always — every user is a doer), Tim (conditional, existing team_tab_state). The new panel carries data-panel="tugas-saya" and is lazy-loaded on first activation, mirroring the Tim tab pattern at vt_home.js:60,73.

The panel renders my_work's three sections verbatim in behavior:

  • My Day — today's schedule-driven tasks; each row carries [Start] and [Submit for Review] action buttons.
  • What To Do Today — high-priority tasks with deadline within 3 days, not blocked.
  • My Blocked Tasks — tasks with active blockers, ordered by days blocked.

Extraction: vt_home.js is already 495 lines; inlining ~130 more worsens a god-file. The panel is extracted to a new shared asset public/js/vt_focus_panel.js exposing window.vt_render_focus_panel(wrapper), wired via hooks.py app_include_js — the same global-asset pattern as public/js/vt_empty.js and public/js/page_nav.js. vt_home.js only calls vt_render_focus_panel() when the Tugas Saya tab is first activated. The three render functions and the two button handlers from my_work.js move into this asset.

4. Backend — move the five functions

When task/page/my_work/ is deleted, the whitelisted methods at vernon_tasks.task.page.my_work.my_work.* die with it. Their logic moves into the existing module task/api/my_work.py (already home to my_work_list; same "my work" domain; function names do not collide). This keeps task/api/dashboard.py (already 1147 lines) from growing further.

FunctionKindBehavior (unchanged)
get_my_dayfetchTasks with a schedule entry for today, excludes DONE phase.
get_what_to_do_todayfetchDeadline ≤ 3 days, not blocked.
get_my_blocked_tasksfetchTasks with active blockers, ordered by days_blocked DESC.
start_taskmutationBacklog/Scheduled → DO/In Progress; refuses if an active blocker exists.
submit_for_reviewmutationDO/In Progress → CHECK/In Review.

The new JS calls vernon_tasks.task.api.my_work.<fn> via a dedicated MW_API const (vt_home.js already uses multiple API namespaces — API for dashboard, ONB_API for onboarding). Functions are moved as-is; no behavior change. Each keeps its doc comment + PRD ref.

6. Tests

  • Retarget task/page/my_work/test_my_work.py (10 cases for get_my_day / get_what_to_do_today / get_my_blocked_tasks / start_task / submit_for_review) to import from vernon_tasks.task.api.my_work; relocate to task/api/test_my_work_focus.py. Behavior assertions unchanged.
  • New tests/test_retire_my_work_patch.py — after patch: Page my-work gone, navbar row with route /app/my-work gone, workspace shortcut gone, vt-home survives. Patch is idempotent (run twice = same result).
  • New vt-home render assertion — Tugas Saya tab present in the tab set.
  • Each test file header tags PRD/ADR IDs it covers; each test function carries an inline PRD-NNN tag.

Tests run inside Docker: docker exec frappe-backend-1 bench --site task.localhost run-tests --module ....

7. Out of scope (YAGNI)

  • No redirect alias for /app/my-work (retire-fully was chosen) — the route 404s after deletion.
  • No changes to the Tim tab, Beranda charts, onboarding, or schedule agenda.
  • No rewrite of api/my_work.py's existing my_work_list / search / mutations.
  • No restyle — Tugas Saya reuses existing .vh-* / my-work-container styles already shipped in vt_home.css.
