Brand Detail Page — OKR per Period

Date: 2026-06-04 Task size: L (new desk Page + 2 API modules + 1-line edit to vt_brands.js + 3 test files + docs; new directory, ~12 files) App ownership: 100% vernon_tasks — no Frappe core files touched Scope: Clicking a brand card on /app/vt-brands opens a new vt-brand-detail desk Page that lists the brand's OKRs grouped by period (newest first), with inline create/edit of Objectives and Key Results via dialogs.

1. Goal

Today /app/vt-brands lists VT Brand records as cards, and clicking a card opens the native VT Brand form. That form shows only brand fields (name, logo, description) — it gives no view of the brand's strategic OKRs. Objectives are scoped to a brand (Objective.brand Link) and carry a period (e.g. 2026-Q1), but there is no per-brand surface to see "what are this brand's objectives this quarter, and how are the key results tracking".

This feature adds a brand detail page that becomes the new click target. It groups the brand's Objectives by period (newest first), shows each Objective's Key Results with progress, and allows inline management (create/edit Objective, create/edit Key Result) through dialogs — without leaving the page. PDCA-phase transitions and deletes stay on the native form (preserve the state machine and cascade guards).

Native Frappe infrastructure only: a desk Page, thin @frappe.whitelist() HTTP entrypoints delegating to the existing DocType controllers. No core files touched.

2. Architecture

2.1 New desk Page vt-brand-detail

Mirrors the established vt-project-detail pattern (hero + body, IIFE-wrapped script, route param).

task/page/vt_brand_detail/
  __init__.py
  vt_brand_detail.json   # Page doc: name vt-brand-detail, roles VT Member/Leader/Manager
  vt_brand_detail.js     # IIFE-wrapped; on_page_load reads frappe.get_route()[1]
  • Route: ["vt-brand-detail", <brand_id>]. The slug vt-brand-detail does not collide with the VT Brand DocType slug vt-brand (see memory: Desk Page vs DocType slug collision).
  • Roles: VT Member, VT Leader, VT Manager (same as vt-brands).
  • Hero: brand logo + brand_name + description; buttons Edit Brand (→ frappe.set_route("Form", "VT Brand", id)) and primary action + Objective (shown only when can_create_objective).
  • Body: one collapsible section per period, ordered by period_start DESC; the "Tanpa Period" bucket renders last. The period whose range contains today (is_current) is auto-expanded; others start collapsed. Section header shows the period label + objective count.
  • Objective card: title, status badge, PDCA phase (read-only label), aggregate progress bar; [edit] + [+ KR] buttons (gated on can_edit).
  • Key Result row: metric, current/target unit, progress bar (progress_percent), confidence; [edit] button.
  • Mutations: all create/edit happen in frappe.ui.Dialog instances; on success the page reloads its data.

2.2 Read API — brand/api/brand_okr.py

Single whitelisted read endpoint. Reuses Objective.get_objective_progress for the aggregate.

@frappe.whitelist()
def get_brand_okr(brand_id: str) -> dict

Response shape:

{
  "brand": {"id", "brand_name", "logo", "description", "can_edit"},
  "can_create_objective": true,
  "periods": [
    {
      "period": "2026-Q1",
      "period_start": "2026-01-01",
      "period_end": "2026-03-31",
      "is_current": true,
      "objectives": [
        {
          "id", "title", "status", "pdca_phase", "owner", "progress",
          "key_results": [
            {"id", "metric", "target", "current", "unit", "progress_percent", "confidence"}
          ]
        }
      ]
    }
  ]
}

Rules:

  • require_login(); frappe.has_permission("VT Brand", "read", brand_id) else PermissionError. Unknown brand → frappe.throw (404-style).
  • Fetch objectives via frappe.get_all("Objective", filters={"brand": brand_id}, ...) ordered period_start DESC, title ASC.
  • Group by the period Data string. Objectives with a blank period go to a trailing "Tanpa Period" bucket (period_start = null, sorts last).
  • is_current = period_start <= today <= period_end (computed per group).
  • can_edit = has_permission("Objective", "write"); can_create_objective = has_permission("Objective", "create").
  • Batch Key Results: one frappe.get_all("Key Result", filters={"objective": ["in", objective_ids]}, ...) then group in Python — no N+1.

2.3 Write API — brand/api/brand_okr_mutations.py

Thin @frappe.whitelist() wrappers that delegate all validation to the DocType controllers via doc.insert() / doc.save(), mirroring task/api/board_mutations.py. Field allow-lists guard against mass-assignment. Native DocType permissions are honored (ignore_permissions=False).

OBJECTIVE_EDITABLE_FIELDS = ("title", "period", "period_start", "period_end",
                             "objective_owner", "status", "description")
KEY_RESULT_EDITABLE_FIELDS = ("metric", "target_value", "current_value",
                              "unit", "confidence")

create_objective(brand_id, values)     # brand FORCED from brand_id, never payload
update_objective(objective_id, values) # verify objective.brand stays consistent
create_key_result(objective_id, values)
update_key_result(kr_id, values)
get_objective(objective_id)            # hydrate edit dialog (editable scalars)
get_key_result(kr_id)                  # hydrate edit dialog

Security / scope rules:

  • pdca_phase is excluded from the objective allow-list — PDCA transitions remain reachable only via the native form, keeping the state machine in Objective.validate authoritative.
  • progress_percent and confidence_last_week are excluded from the KR allow-list — progress_percent is controller-computed; confidence_last_week is system-managed.
  • create_objective sets brand = brand_id from the path param, not from values — an objective created on a brand's page always belongs to that brand.
  • Each mutation re-checks frappe.has_permission for the create/write operation before touching data.
  • values is parsed defensively (JSON string or dict), like portal_brands._parse_payload.

2.4 Client wiring — vt_brands.js (1-line change)

// before
card.on("click", () => frappe.set_route("Form", BRAND_DOCTYPE, b.id));
// after
card.on("click", () => frappe.set_route("vt-brand-detail", b.id));

3. Data flow

vt-brands card click
  └─> route ["vt-brand-detail", brand_id]
        └─> on_page_load → get_brand_okr(brand_id)
              └─> render hero + period sections + objective cards + KR rows

[+ Objective] / [edit objective]
  └─> Dialog → create_objective / update_objective → controller validate → reload

[+ KR] / [edit KR]
  └─> Dialog → create_key_result / update_key_result → controller validate
        (progress_percent auto-computed) → reload

4. Error handling

  • Unknown / no-permission brand: read endpoint throws; page renders the shared empty state ("Brand tidak ditemukan").
  • Validation errors (period grammar, target_value <= 0, illegal PDCA, disabled owner): surface from the controller as frappe.throw; the dialog shows the message and stays open so the user can correct input.
  • No objectives: page renders hero + an empty-state card ("Belum ada OKR untuk brand ini").
  • Permission-limited user: can_edit=false hides edit/create buttons; the read view still works for VT Member.

6. Out of scope (YAGNI)

  • PDCA phase transitions on this surface — stay on the native Objective form (preserves the Deming state machine).
  • Deleting Objectives / Key Results inline — stay on the native form (preserves the on_trash cascade guard).
  • KPI Definition / KPI Entry management — separate OKR sub-domain, not part of the period-grouped view.

These can be added later without reworking the page shell or API shape.
